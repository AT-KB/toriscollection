## 1. Android版「初回起動で白紙固まり」問題 — 解決済み・本番反映済み

**根本原因**: `render_login_screen()`(初回ユーザーが最初に見る「新規スタート/セーブコード
再開」の選択画面)で、ネイティブスプラッシュを消す `_inject_splash_hide()` が一度も
呼ばれていなかった。画面自体はDOM上完全に描画され`はじめる`ボタンまで押せる状態だったが、
消えないスプラッシュに永久に覆われて誰にも見えなかった。CORS・WebSocket圧縮・User-Agent・
Renderコールドスタートは全部無関係(実機のChrome DevTools Protocolで直接DOM状態を見て確定)。

**修正**: `toris_collection/app.py` の `render_login_screen()` の `st.stop()` 直前で
`_inject_splash_hide()` を呼ぶよう1箇所追加(commit `f204557`)。

**現在の状態**: **Renderにデプロイ済み、既に本番で全ユーザーに反映されている。**
このアプリはCapacitor WebViewが `https://toris-collection.onrender.com` をライブで
読み込むだけの構造なので、この修正はPlay Storeの更新を経ずに即座に効く
(実機・本番URLで動作確認済み)。

### 副産物: MainActivity.java(ネイティブ側)の修正はまだPlay Store未提出

調査の過程で `android_app/android/.../MainActivity.java` にも以下を追加済み(いずれも
無害・commit済みだが、**現在公開中のPlay Storeバージョン(versionCode 2)にはまだ
含まれていない**):
- watchdogタイムアウトを25秒→90秒に延長
- 「一定回数リロードすると監視を完全に諦めて二度と回復しない」バグの修正(無期限バックオフ化)
- User-Agentの"; wv)"マーカー除去(仮説3、これは既にversionCode 2で提出済み)
- `LAYER_TYPE_SOFTWARE`(GPUタイルメモリ警告への対処、効果はなかったが無害なので残置)

**残存リスク**: 現行Play Store版(versionCode 2)は、Renderのコールドスタートが75秒
(25秒×3)を超えて長引いた場合に「watchdogが永久に諦める」旧バグをまだ踏む可能性がある。
根本原因(スプラッシュ消去漏れ)自体はサーバー側で直っているので致命的ではないが、保険として
新しいAABを用意した:

- **`versionCode 3` にバージョンアップ済み(commit `39b63b2`)、AABビルド済み**:
  `android_app/android/app/build/outputs/bundle/release/app-release.aab`
- **CEOの判断で今回の提出は見送り。** 提出したくなったら、Play Console →
  対象アプリ → リリース作成 → 上記AABをアップロードするだけでよい(ビルドし直す必要はない。
  ただし他のコード変更が入った場合は再ビルド要)。

## 4. BGMボタンの体感が薄い件 — 原因特定、対応は保留中

**原因**: 本番でFreesound APIキーが未設定だと、ヒーリングBGMの環境音(パッド)が
実録音ではなく合成の和音+微弱ノイズにサイレントにフォールバックする
(`freesound_client.py`)。しかもこの合成パッドはBGM ONでも音量ゲイン0.11までしか
上がらず、本来の実録音版(0.28)の半分以下の音量にしかならない。

**現状**: CEOがFreesoundでAPIキーを取得し、Renderの環境変数 `FREESOUND_API_KEY` に
設定済み(「設定しました」と報告あり)。ただし**このセッション内では実際に効いているかの
直接確認ができていない**(検証しようとしたが、ラジオの音声コンポーネントは観察済みの鳥が
0羽だと描画されない仕様のため、新規セッションでは検証不可だった)。

**次に確認すべきこと**: Renderダッシュボード → 対象サービス → Logsタブで、直近の再起動ログに
```
[freesound] APIキー読み込み済み(末尾4文字: ...xxxx)
```
が出ているか確認する。まだ
```
[freesound] APIキー未設定。環境音はシンセ合成にフォールバックします。
```
が出ている場合は、環境変数名(`FREESOUND_API_KEY`)や値の設定ミスを疑う。

キーが有効化されていることを確認できたら、企画部提案(§3)の「切替の瞬間の速い時定数」
「st.toast()での即時フィードバック」等の追加改善が必要か、CEOに体感を確認してもらう。
CEOからは「(現状の)緩やかな変化はいいよ」というコメント済みなので、音量差そのものを
広げる系の改善は急ぐ必要はなさそう。
