# PWA化(TWA準備)調査・実装結果(2026-07-05・開発部)

Google Play ロードマップ Phase 0/Phase 1向け下準備。`ROADMAP_GOOGLE_PLAY.md`
(2026-07-05改訂・TWA先行方針)に基づく、Streamlit上での最小PWA品質の追加。

## 結論(先に要点)

- **web app manifest は実装・実ブラウザ確認済み**。Bubblewrap(TWA生成)が
  読みに行く `manifest.json` は正しく配信され、トップレベルの `<head>` にも
  `<link rel="manifest">` と `<meta name="theme-color">` を実際に注入できる
  ことをPlaywrightで確認した。
- **サービスワーカーは「用意はしたが登録していない」**。Streamlitの静的ファイル
  配信の仕様上の制約により、`.js` ファイルは常に
  `Content-Type: text/plain` + `nosniff` で配信され、ブラウザは
  ServiceWorkerとして登録できない(実機で `unsupported MIME type` エラーを
  確認済み)。これはこちらのコードでは回避できない、Streamlit側の設計上の
  制約。詳細は下記。
- 大掛かりな作り直しは発生していない。追加は「静的ファイル3つ +
  config.toml 1行 + app.py に1関数呼び出し」のみで、既存のタブ構成・コアループ・
  ラジオ・儀式には触れていない。

## 調査したこと

### 1. Streamlit標準機能でどう実現するか

- Streamlitは `<head>` への直接注入手段を持たない。`st.markdown(unsafe_allow_html=True)`
  で挿入したHTMLは `dangerouslySetInnerHTML` 相当でDOMに挿入されるだけで
  `<script>` は実行されない(ブラウザの一般的な仕様)。
- 一方 `components.html()` はソースコード上 `sandbox="allow-forms allow-modals
  allow-popups allow-popups-to-escape-sandbox allow-same-origin allow-scripts
  allow-downloads"` を持つ `srcdoc` iframe を生成する
  (`streamlit/static/static/js/IFrameUtil.*.js` を確認)。`allow-same-origin`
  + `allow-scripts` の組み合わせにより、iframe内スクリプトは
  `window.parent.document` を操作できる(実機で動作確認済み。ブラウザは
  コンソールに "can escape its sandboxing" という警告を出すが、これは
  Streamlit自身のcomponents機構が既存の `radio.py`/`ritual.py` でも使っている
  枠組みであり、今回新たに広げたリスクではない)。
  → この仕組みを使い、`components.html(js, height=0)` から
    `parent.document.head.appendChild()` で `<link rel="manifest">` と
    `<meta name="theme-color">` を注入する実装にした。
- **静的ファイルの配信**: Streamlit 1.53 は `server.enableStaticServing = true`
  (`.streamlit/config.toml`)を設定すると、メインスクリプトと同じ階層の
  `static/` フォルダを `<origin>/app/static/<path>` で配信する
  (`streamlit/file_util.py: get_app_static_dir` / `APP_STATIC_FOLDER_NAME`)。
  公式のドキュメント化された機能であり、大掛かりな仕組みは不要だった。

### 2. サービスワーカーが登録できない理由(実機確認で判明)

`streamlit/web/server/app_static_file_handler.py` の
`AppStaticFileHandler.set_extra_headers()` は以下の実装になっている:

```python
SAFE_APP_STATIC_FILE_EXTENSIONS = (
    ".jpg", ".jpeg", ".png", ".gif", ".webp",
    ".otf", ".ttf", ".woff", ".woff2",
    ".pdf", ".xml", ".json",
)
# 上記以外の拡張子は問答無用で Content-Type: text/plain
```

`.js` はこの許可リストに含まれていない。実際にローカルでStreamlitサーバーを
起動し、Playwright(既存Chromium、`executable_path` 指定)で
`navigator.serviceWorker.register('/app/static/sw.js', {scope: '/'})` を
実行したところ、以下のエラーで**必ず**失敗することを確認した:

```
Failed to register a ServiceWorker for scope ('http://localhost:8511/')
with script ('http://localhost:8511/app/static/sw.js'):
The script has an unsupported MIME type ('text/plain').
```

`X-Content-Type-Options: nosniff` も同時に付与されるため、拡張子を変えたり
別の場所に置いたりしても、Streamlitの静的配信を経由する限りこの制約からは
逃れられない。回避するには次のいずれかが必要になり、いずれも今回のスコープ
(壊さない・最小追加)を超えるため、**実装せずここで報告に留める**:

- Streamlit本体のパッチ/フォーク(壊さない方針・「今のものをベースに」に反する)
- 別ホスティング(小さな静的サーバーやCDN)を1ファイルのために追加
  (「サーバー・運用の負荷を増やしたくない」というCEO意向に反する)
- Streamlit本体への機能要望(`.js`を許可リストに追加してもらう)を出し、
  将来のアップデートを待つ

### 3. TWA生成そのものに実際どこまで必要か

再確認: **Bubblewrap(TWA生成ツール)は `manifest.json` のURLさえ渡せば
Android プロジェクトを生成できる**。サービスワーカーはブラウザの
「インストール可能なPWA」判定(Chromeの追加ホーム画面バナー等)や、Play
ストアの推奨品質基準(オフライン時の白紙防止)には関わるが、Bubblewrap
のビルド自体の必須要件ではない。したがって:

- 今回実装した manifest.json 単体でも、**Phase 0/1のTWA生成作業には着手できる**。
- サービスワーカーが無いことで生じる影響は「オフライン時にChromeの恐竜画面が
  出うる」という体験面の劣化のみで、Google Play申請のブロッカーではない
  (Bubblewrap公式ドキュメントもサービスワーカーを必須要件として扱っていない)。

## 実装した内容

- `toris_collection/static/manifest.json`(新規): name/short_name/description、
  `start_url: "/"`、`scope: "/"`、`display: "standalone"`、
  `background_color: "#f7f9f5"`、`theme_color: "#7ba87b"`
  (既存UIの配色 `app.py` の `<style>` ブロックに合わせた)。
  アイコンは既存 `designbird/mejiro.png`(64x64のドット絵)を
  ニアレストネイバー法で192x192/512x512に拡大したもの
  (`toris_collection/static/icons/icon-192.png` / `icon-512.png`)。
  ドット感を保ったまま拡大されており、専用絵に後で差し替え可能
  (`SPRITE_ALIASES` の流用方針と同じ考え方)。
- `toris_collection/static/sw.js`(新規): install/activate/fetchのみの
  最小サービスワーカー。**現時点では未登録**(上記の理由)。将来
  Streamlit側の制約が解消された場合、または別ホスティングを承認する
  場合に備えて実体だけ残置。
- `.streamlit/config.toml`(新規、リポジトリルート): `server.enableStaticServing
  = true`。Streamlit Community Cloudはリポジトリルートを作業ディレクトリとして
  `streamlit run toris_collection/app.py` を起動する認識のため、ここに設置。
- `toris_collection/.streamlit/config.toml`(新規、同内容の複製):
  README記載のローカル起動手順(`cd toris_collection && streamlit run app.py`)
  では作業ディレクトリが `toris_collection/` になるため、cwd依存を
  ヘッジする目的で同内容を複製した。
- `toris_collection/app.py`: `_inject_pwa_head()` を追加し、
  `st.set_page_config` 直後(スタイル注入の直後)に呼び出し。
  `components.html()` 経由で `parent.document.head` に
  `<link rel="manifest">` と `<meta name="theme-color">` を注入するのみ。
  try/exceptで握りつぶすため、将来Streamlit内部構造が変わって失敗しても
  本編(ラジオ・儀式・図鑑・不在ループ等)には影響しない。

## 壊さない検証(実施結果)

1. **構文チェック**: `app.py` を含む変更ファイルすべて `ast.parse` でエラーなし。
   `static/manifest.json` は `json.load` で有効なJSONと確認。
   `static/sw.js` は `node --check` で構文エラーなしと確認。
2. **AppTestで起動時例外0**: `AppTest.from_file("app.py").run()` →
   `at.exception` は空(`ElementList()`)。
3. **tests全pass**: `tests/test_*.py` 全6ファイル、47ケース全てpass
   (community 7 / daily 7 / disturbance 5 / feeder_chain 8 / flock 8 /
   save_code 12)。今回はJS/PWA領域のため純粋関数の新規テストは追加していない
   (対象が「ブラウザのDOM/Service Worker API」であり、stdlibのみの
   ユニットテストに落とし込めないため、4番目のPlaywright確認で代替)。
4. **ラジオJS/実挙動確認(Playwright)**: ローカルで実際に
   `streamlit run app.py --server.enableStaticServing true` を起動し、
   既存Chromium(`executable_path` 指定、再ダウンロードなし)で確認。
   - `GET /app/static/manifest.json` → 200、`Content-Type: application/json`
   - `GET /app/static/icons/icon-192.png` / `icon-512.png` → 200、`image/png`
   - `GET /app/static/sw.js` → 200(ただし `Content-Type: text/plain` のため
     未登録の判断根拠となった)
   - トップドキュメントの `document.querySelector('link[rel="manifest"]').href`
     → `.../app/static/manifest.json`(注入成功)
   - `document.querySelector('meta[name="theme-color"]').content`
     → `#7ba87b`(注入成功)
   - サービスワーカー関連のコンソールエラーは0件(登録処理自体を削除したため)
   - ページタイトル・本文が正常に描画され、既存機能への影響を示す
     クラッシュ・例外は観測されなかった

## 承認が必要な事項

今回の実装自体(manifest.json配信 + `<link>`/`<meta>`注入)は
「軽微なUI調整・既存方針の範囲内」と判断し、実装まで完了させた
(壊さない検証も全てpass)。ただし、以下は判断が分かれる可能性があるため
明示しておく。

- **サービスワーカーを実際に機能させたい場合**、Streamlit標準機能の範囲では
  不可能であり、別ホスティング/Streamlit本体改変等のインフラ変更が必要になる。
  これは「サーバー・運用の負荷を増やしたくない」というCEO意向と直接
  トレードオフになるため、**今回は実装せず、必要ならCEO判断を仰ぐ**。
- アイコンは `designbird/mejiro.png` を流用したプレースホルダ。
  ブランドとして正式なアプリアイコンにするかは企画部/CEO判断。
