# TWA品質事前確認(Lighthouse/サービスワーカー要件の切り分け)(2026-07-05・開発部)

Google Play ロードマップ Phase 1(TWA化)着手前の追加確認依頼への回答。
実装は行わず、Googleの現行ドキュメント(一次情報)を確認した調査報告のみ。

## 結論(先に要点)

- **Lighthouse の「PWAスコア」という概念自体、現在は存在しない。** Google は
  2022年後半(Chrome 108/mobile・112/desktop)以降、インストール可否判定から
  「fetch()を実装したサービスワーカー」の要件を撤廃し、それに伴い
  **Lighthouse から PWA カテゴリそのものを廃止した**(理由: 空のfetchハンドラを
  置くだけで基準を満たす"チェックボックス埋め"が横行し、実効性がなかったため)。
  したがって「何点必要」という基準は、Bubblewrap側にもPlay側にも実体として存在しない。
- **Bubblewrap(TWA生成)は `manifest.json` のURLさえ読めればプロジェクトを
  生成・ビルドできる。** Lighthouseスコアやサービスワーカーの有無を検査する
  ゲートはBubblewrap自身のツールチェーンに存在しない(README・Quick Start Guide
  で確認済み)。
- **サービスワーカーが無いこと自体はPlay審査を機械的に弾く要件ではない。**
  ただし、Chrome 86 以降の「TWA品質基準」(現行ドキュメントにも記載が残る)では、
  ①デジタルアセットリンク検証失敗 ②オフライン時にリソースがHTTP 200を返せない
  ③ナビゲーションが404/5xxを返す、の3パターンを適切にハンドルしないと
  **Android上で"ユーザーに見えるクラッシュ"として扱われる**。②③はサービス
  ワーカーで緩和することが公式に推奨されている。これは「審査で弾かれる」話
  ではなく、公開後のPlay Console「クラッシュ率(vitals)」に計上されうるという
  運用上のリスク。今回の規模(クローズドテスト12人×14日)ではこの閾値に
  到達する可能性は低いと判断する。
- **別ホスティング(GitHub Pages等)にsw.jsだけ置く案は、技術的に不可能**
  (前回調査では「大掛かりだから見送り」という運用上の理由だったが、実際は
  ブラウザの仕様上そもそも動かない)。Service Worker の `register()` は
  スクリプトURLとページが**同一オリジンでない場合は `SecurityError` で失敗する**
  仕様であり、Streamlit Cloud と別ドメインのGitHub Pages を跨ぐ構成は原理的に
  実現できない。よって「軽量な回避策」は存在せず、**対応不要**の結論を補強する。
- **判定: Phase 1のTWA生成にこのまま進んでよい。追加対応は不要。**

## 調査したこと(一次情報)

### 1. Lighthouse PWAカテゴリの現状

`developer.chrome.com/blog/update-install-criteria`(最終更新 2023-12-05)より引用:

> In the coming months Chrome will be running experiments with the goal of
> simplifying the app install criteria... As a first step we have removed the
> requirement to have a service worker that implements the fetch() method for
> installation from the menu, since version 108 on mobile and 112 on Desktop.
> ...because the Lighthouse PWA checks are directly associated with the
> installability criteria, we have decided to remove this category from
> Lighthouse.

`developer.chrome.com/docs/lighthouse/overview`(最終更新 2025-06-02)の説明文
(「Lighthouse has audits for performance, accessibility, SEO, and more」)にも
PWAカテゴリの言及は無く、現行のLighthouseレポートにPWAスコアという項目は
存在しないことと整合する。

`web.dev/articles/install-criteria`(最終更新 2024-09-19)の現行インストール
基準一覧にも、サービスワーカーは**含まれていない**(HTTPS・manifestのname/
icons/start_url/display等のみ)。

→ 「Lighthouseスコアが何点必要」という基準そのものが、現在は実体として
存在しない。混同すべきでないのは「Bubblewrapの必須要件」と「(既に廃止された)
Lighthouse PWAバッジ」の2つであり、今回確認した限り両方とも実体としては
存在しない/最小限(manifestのみ)。

### 2. Bubblewrap(TWA生成)自体の必須要件

`developer.chrome.com/docs/android/trusted-web-activity/quick-start` より:

> bubblewrap init --manifest=https://my-twa.com/manifest.json

manifest.jsonのURLを渡すだけでプロジェクトが生成できる。Bubblewrap README
(`GoogleChromeLabs/bubblewrap`)には `bubblewrap/validator` という
「PWAが品質基準を満たすか検証するライブラリ」が存在するが、これは**任意の
検証ツール**であり `init`/`build` の実行を妨げるゲートではない。

### 3. サービスワーカーが無い場合に実際何が起きるか

`developer.chrome.com/docs/android/trusted-web-activity/whats-new`
(最終更新 2020-12-02、現行ナビゲーションに掲載され続けている生きたドキュメント)より:

> Chrome 86 introduced changes to the Trusted Web Activity Quality Criteria...
> you should ensure your applications handle the following scenarios to
> prevent them from crashing:
> - Failure to verify digital asset links at application launch
> - Failure to return HTTP 200 for an offline network resource request
> - Return of an HTTP 404 or 5xx error in the application.
> Besides ensuring that the application passes the Digital Asset Links
> validation, the remaining scenarios can be handled by a service worker.

`web.dev/articles/using-a-pwa-in-your-android-app` でも同じ3シナリオが
「user visible crash」として説明されている。

つまり:
- **Bubblewrap生成・Play提出(アップロード)は妨げられない**(ビルドプロセスに
  Lighthouse/SWの自動チェックは組み込まれていない)。
- **実機で起きるのは「Playの審査落ち」ではなく「Android上のクラッシュ扱い」**。
  具体的には、オフライン時にTWAを開く/操作すると、ブラウザのオフラインページ
  ではなく、Androidアプリとしての"クラッシュ"としてカウントされうる
  (Play Console の Android vitals にクラッシュ率として反映されるリスク)。
- Toris Collection はそもそも xeno-canto 音声取得・Google Sheets 等ネットワーク
  必須の構成であり、オフラインで正常動作しない前提は既に織り込み済み
  (`ROADMAP_GOOGLE_PLAY.md` の「壁1」参照)。SWの有無で追加的に生まれる実害は
  「オフライン時の見え方がブラウザのエラーページかAndroidのクラッシュ画面か」
  という程度の差分。
- クローズドテスト規模(12人×14日)は非常に小さく、Play Consoleの「不良な
  動作」判定が一般に適用される規模(相当数のインストール・セッション数を
  前提とする)に達する可能性は低い。**Phase 3(製品版申請・DL数拡大)時点で
  改めてクラッシュ率を監視する**運用でよい。

### 4. 別ホスティング(GitHub Pages等)でsw.jsだけ配信する案の可否

`developer.mozilla.org/en-US/docs/Web/API/ServiceWorkerContainer/register`
の例外一覧より:

> SecurityError: ... The scriptURL and scope are not same-origin with the
> registering page.

Service Worker の仕様上、**登録元ページとスクリプトURLは同一オリジンである
必要があり**、別オリジン(例: GitHub Pages)に置いたスクリプトを
Streamlit Cloud上のページから `register()` することは**ブラウザによって
拒否される**。前回調査(`2026-07-05_PWA化調査.md`)ではこの案を「サーバー
負荷が増えるため見送り」という運用判断で退けていたが、実際には
**そもそも技術的に成立しない**ことが確認できた。したがって「軽量な回避策」
は存在せず、CEO方針(サーバー運用負荷を増やしたくない)を持ち出すまでもなく
**対応不要**と判断してよい。

## 判定

**Phase 1のTWA生成(Bubblewrapによる `manifest.json` からのAndroidプロジェクト
生成・クローズドテスト提出)にこのまま進んでよい。** サービスワーカー未登録を
理由に追加対応を挟む必要はない。

念のため付記する監視事項(今回のスコープの「実装」ではなく、将来の観察ポイント):

- Phase 1提出後、Play Console の Android vitals で「ANR/クラッシュ率」に
  異常値が出ないかを一度確認する(オフライン起動由来の擬似クラッシュが
  多発していないかの確認。クローズドテスト規模では起きにくいと予想するが
  実測は無料でできる)。
- Phase 3(製品版申請・DL数拡大)で実際にクラッシュ率が問題になった場合に
  初めて、サービスワーカー実装(Streamlit本体の制約回避)を再検討すればよい。
  現時点では**対応不要**。
