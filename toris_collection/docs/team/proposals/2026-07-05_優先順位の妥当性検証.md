## 1. 判断軸の再検証:「ラジオを太くする」は既に2つのエンジンで稼働している

CEO指摘とHANDOFF §1-1の規律(「ラジオを目に見えて豊かにするか/ユーザーを引き戻すかを満たすこと」)
に照らして、**まず現状を正確に把握する**。結論から言うと、この規律を満たす施策はすでに
**2つ実装・稼働済み**であり、「これから太くする」のではなく「既に太くなっている実態を、
どう見せるか」が今回の1〜3の実質的な争点である。

- **`disturbance.py`(撹乱・遷移)**: 倒れた跡地に別種が芽吹き(`roll_succession`、exclude で
  同種再生を防止)、**新しい顔ぶれをラジオに供給する**エンジン。HANDOFF §4-3直前(L200-217)に
  「合格」と明記済み、実装済み。
- **`flock.py`(群れ)**: 観察回数(familiarity)に応じて既存の鳥の声を「厚くする」
  (`flock_size`、`GROWTH_EVERY=3`回ごとに+1)。HANDOFF §4-3(L219-234)で「合格」と明記済み、
  実装済み。
- **`absence_loop.py`の復帰フック**(HANDOFF §4-4 L240-242): 留守中の初到来を
  `radio_new_arrivals` に積み、🌟バナーで「ラジオに加わりました」を可視化する仕組みも
  **既に実装済み**。

つまり「会いに行く→ラジオの仲間が増える」という背骨のループは、供給側(撹乱)・厚み側(群れ)・
通知側(復帰フック)の3点がすでに揃っている。CEOの問い(「太くすることが最優先では」)は
正しい優先軸だが、**ゼロから太くする機能を足す段階ではなく、既にある豊かさへの
"気づき"を強める段階**、というのが企画部としての現状認識である。この認識は、後述する
1〜3の評価(特に③生態ログの重要度の見直し)に直結する。

## 2. おやつ(餌台)配線の実態検証 — コードを読んだ結果、想定より重く・効果が限定的

「おやつ配線がラジオを太くする最優先施策」という部長の追加提案を、コードで裏取りした。
**結論: 優先順位に置くこと自体は妥当だが、想定されている効果と実装スコープの両方に
訂正が必要。** 以下、確認した事実。

### 2-1. 「置くと来る鳥が増える」効果は、現状のコードには存在しない

`feeder_chain.py`を全文確認した。実装されているのは以下の一方向のみ:

```
開放型の餌台 → リス(gray_squirrel)→ クーパーハイタカ(cooper_hawk)
                                    → wary_arrival_multiplier() で
                                      警戒心の強い鳥の到来確率を「抑制」
```

`wary_arrival_multiplier(wariness, raptors)`(L93-103)は`1.0 - strength * w`という**減算のみ**の式で、
どの鳥の到来確率も**上げる項が一切ない**。`engine.py`の食物網グラフ(`build_network`、L196-240)は
`bird["eats_plants"]`/`bird["eats_insects"]`のみからエッジを張っており、`feeder_open`のような
給餌ノードはグラフに一切登場しない(feeder自身がグラフのノードやエッジになる経路が無い)。

一方、`CONCEPT_v2.md`(L118)は「役割: 来る鳥の"多様性・意外性"を少し上げる(**レア来訪の確率↑**)」
と明記しており、これは**現状のコードでは実現されていない**。実装すると、ユーザー体験として
起きるのは「餌台を置いても特に何も増えない。むしろ稀にハイタカが来て、臆病な鳥が少し来にくくなる」
という**抑制のみの体験**になる。これは生態的にはリアルで面白い駆け引き(実際のバックヤード
バーディングにある現象)だが、**CEOが問う"ラジオの仲間を増やす"効果には直接寄与しない**。

### 2-2. `placed_features`という状態自体が、コードのどこにも存在しない

`engine.py`・`app.py`を含む全体を`placed_features`でgrepしたが、`feeder_chain.py`・
`tests/test_feeder_chain.py`・docsの記述以外に**実装は一切ない**。つまり「UI配線」という
言葉が示唆する規模感(=既にある内部状態にUIを足すだけ)は不正確で、実際には
以下をゼロから作る必要がある:
1. `st.session_state`に新規キー(例: `placed_features`)を追加。
2. `run_turn`/`calculate_arrival_probability`(`engine.py` L243-345)に
   `wary_arrival_multiplier`を掛ける引数を新設(現状シグネチャにその余地がない)。
3. **セーブコード(`2026-07-04_ローカル保存MVP技術検討.md`で確定した`save_code.py`の
   永続化対象キー一覧、`biome`/`planted`/`residents`/`discovered`/`bird_days`/`mementos`/
   `bird_notes`/`observed`)に`placed_features`が含まれていない**。これを足さないと、
   セーブコードで復元するたびに置いたおやつが消える。

以上より、実装規模感は前回提案書の「小〜中規模」ではなく**中規模**に訂正すべきである
(開発部への申し送り事項)。

### 2-3. 推奨(スコープ修正案。優先順位そのものの否定ではない)

おやつ配線を優先順位2として進めること自体は妥当(既存ロジックの再利用・無料資源運用・
生態に誠実という前回チェックは変わらず有効)。ただし以下2点の**スコープ修正**を提案する:

1. **訴求文言を「駆け引き」に留め、「レア来訪が増える」と謳わない**。実装されるのは
   抑制側のみのため、ユーザー向け説明は「開放型の餌台はリスを呼び、まれにハイタカも
   訪れます。ハイタカが来ている間は、臆病な鳥は少し来にくくなります」という**正直な
   記述**にする。過大な期待をさせると「置いたのに何も変わらない」という
   Unmet Need #1(ねこあつめ型の苛立ち)を自ら再生産するリスクがある。
2. **リス・ハイタカの出没を③生態ログに一言残す**(後述§3で優先度を再評価する生態ログの
   データ構造をそのまま使う)。図鑑・会った日数には入れない(既存方針どおり)が、
   「今日はリスが庭に来ていました」のような**揮発しない記録**を1箇所作ることで、
   おやつを置いた"手応え"を担保する。これにより②と③は独立した2施策ではなく、
   **③のデータ基盤の上に②が乗る1本の設計**にできる。

## 3. 抜け漏れ検討: 生態ログの優先度を③のまま据え置いてよいか → 上げるべき

§1の再検証(撹乱・群れは既に稼働中)を踏まえると、**生態ログ(現③)こそが
「太くする」を最も直接に可視化する施策**であるという評価に至った。理由:

- 撹乱・群れは**確率エンジンとしてはすでに機能している**が、その結果(なぜ多様になったか)は
  `absence_loop.build_reason_text()`が生成した瞬間にバナー表示で消える(揮発性)。
  「遊ぶほど生態系がわかる」(`00_共通サマリ.md`一行目)という一行サマリーが、
  実は**すでに生成されているのに、蓄積される場所がない**状態。
- 生態ログを先に(または②と同時に)作ることで、②おやつの成果(リス/ハイタカの出没)も
  ①バッジの成果(節目の記録)も、**同じ「積み上がる記録」という文法**に乗せられる。
  逆に②を先に作ると、おやつの出来事だけが行き場を持たない一過性の演出になり、
  後から生態ログに接続し直す手戻りが発生しうる。

**提案**: ①バッジ→③生態ログ→②おやつ、の順に組み替えるか、少なくとも
②と③を同時並行で設計する(③のデータ構造を先に固め、②はそこに乗せる)ことを推奨する。
これは「1〜3を今回実施」という大枠(範囲・原則)を変えるものではなく、**着手順の
入れ替え・依存関係の明示化**に留まる提案であり、CEO確認は不要と判断する
(範囲・原則に変更はない)。

## 4. 抜け漏れ検討: 「今日の庭」(daily.py)の磨き込みは含めるべきか → 小さく含めるべき

**結論: 含めるべき。ただし新規タブ・新規メカニクスにはせず、既存カードの表示強化のみに
スコープを絞る。**

根拠:
- 市場: Merlin Bird IDの「今日の発見フィード」的パターン(日替わりの1件をハイライトする
  設計)は、`daily.py`が**既に日付+土地の決定的シードで実装済み**(`todays_bird`、L28-42)。
  ただし現在の見せ方(`render_todays_garden`、L53-93)は**テキストと色付きの帯のみ**で、
  ドット絵も鳴き声プレビューも出していない。
- 社内資産: `DIRECTION_BUSINESS_MODEL.md`(L93)「報酬の瞬間を磨く: 『今日の庭に◯◯が来た』を、
  飛来アニメ+鳴き声+1枚カードで祝う」という部長提言は、当時は一般的な到来演出への言及だったが、
  現在`daily.py`という同名の具体機能が存在するため、**そのまま適用できる具体的な改善対象**になっている。
- コスト: `daily.py`自身の鳥ID(`bid`)から`birds_data[bid]`経由でドット絵・音源は
  **既存データをそのまま参照可能**(新規アセット・新規ロジック不要)。①バッジと同様、
  「既存データの見せ方を変えるだけ」で完結する低コスト施策。

**線引き(daily.py自身の設計思想を壊さない)**: `daily.py`冒頭コメント(L11)
「別画面のクイズにはしない(やり過ぎるとラジオが死ぬ)」を厳守する。追加するのは
①カードにドット絵サムネイルを添える、②タップ/クリックで鳴き声を試聴できる
(声のアクセスは元から無料なので原則3に抵触しない)、の2点に限定し、
新規タブ化・ゲーム化・連続日数条件の追加は一切行わない。

**優先順位への位置づけ**: ①バッジ(図鑑カードの演出)と技術的性質が同じ
(既存データ+表示強化のみ)ため、①と同時に着手できる**並走可能な小タスク**として
4番目に追加することを提案する。1〜3の範囲・原則を変えるものではないため、
CEO確認は不要と判断する。

## 6. まとめ(推奨アクション)

1. ①バッジ・③生態ログはそのまま進めてよい(前回チェック済み・追加リスクなし)。
2. ②おやつ配線は優先順位に残すが、**(a)実装規模感を「中規模」に訂正**
   (`placed_features`新設・`engine.py`シグネチャ変更・`save_code.py`スキーマ追加が必要、
   単なる「UI配線」ではない)、**(b)訴求文言を「駆け引き」に限定**(現状のコードには
   到来確率を上げる経路が無く、抑制のみのため)、**(c)出没イベントを③生態ログに記録する
   設計にする**、の3点を開発部へ申し送ることを推奨する。
3. 着手順は①→③→②(または③②同時並行)への組み替えを推奨する。③は撹乱・群れという
   既に稼働中の「太くする」エンジンの成果を可視化する最短距離であり、②のデータ受け皿にもなる。
4. 「今日の庭」の磨き込み(ドット絵+鳴き声プレビューの追加のみ、新規タブ化はしない)を
   ①と並走できる4番目の小タスクとして追加することを推奨する。
5. 上記いずれも`CONCEPT_v2.md`の★★★核機能表やHANDOFF §8の確定済み設計判断を覆すものではなく、
   既存の1〜3の範囲・原則を維持したままの**着手順の入れ替えとスコープの精緻化**に留まるため、
   **CEO確認は不要**と判断する。ただし②の実装規模の訂正(小〜中→中)は、部長経由で開発部との
   工数合意の際に共有すべき。

### 結論(300字程度)

1〜3の大枠(範囲・原則)は妥当。ただしコード検証の結果、②おやつ配線は「UI配線のみ」ではなく
`placed_features`新設・`engine.py`シグネチャ変更・セーブコードのスキーマ追加を要する中規模作業であり、
また現行`feeder_chain.py`は警戒鳥の抑制のみでレア鳥を増やす経路がないため、「来る鳥が増える」と
訴求しないこと。撹乱・群れは既に稼働中の「太くする」エンジンであり、③生態ログはその成果を
可視化する最短距離のため、優先度を②と同等以上に引き上げ、②の出没イベントは③に記録する設計に
一本化することを推奨する。加えて「今日の庭」へのドット絵・鳴き声プレビュー追加を、①と並走する
小タスクとして4番目に加えるべき。いずれもCEO確認は不要。
