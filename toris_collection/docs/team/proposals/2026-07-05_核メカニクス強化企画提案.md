## 1. Unmet Needs 3点の判定(部長仮説を検証)

### 1-1. 待ち時間・運への苛立ち(ねこあつめ) — **構造的には解決済み、ただし2つの残存リスクあり**

**解決済みと判断する根拠**:
- ねこあつめの苛立ちの本質は「特定のアイテムを置く/用意する"コスト"を払ったのに、狙った猫が
  来ない」という**投資と結果の乖離**にある。Torisは `absence_loop.evolve_state()` が示す通り、
  到来は**時間経過のみで自動進行**し、プレイヤーは到来のたびに何かを消費しない
  (`CONCEPT_v2.md`のコアループ「離れる→時間が進む→鳥が現れて来ている」)。**失うものがないので、
  外れても"損をした"感情の土台がない**。原則2(罰しない)の精神と自然に一致している。
- `mementos.py` の設計もねこあつめと決定的に違う。落とし物(twig/feather/plume)は**鳥の来訪・鳴き声・
  発見という主報酬の"おまけ"**であり(`discovered` は到来だけで確定、`app.py:859`)、
  ガチャの当落そのものが体験の主目的になっていない。「ハズレ」でも主報酬(声・かわいさ)は必ず得られる
  ——ねこあつめの"置いたのに来ない"とは構造が違う。
- `daily.py` の「今日の庭」が、確率とは独立に**その日必ず存在する具体的な目標**(今日の一羽)を
  与えており、"何をしても無駄"という無力感を緩和している。

**未解決・残存リスク(2点)**:
1. **完全コンプリート狙いのユーザーには依然運の変動が残る**。`data.py` の `rarity` は最大0.75、
   `INTERNAL_SPEC.md` §4 の `rarity_factor = (1.0 - rarity×0.85) × 0.9` に加え、`mementos.py` の
   `plume` 落下率4%(`PLUME_BIRDS` 該当種のみ)が重なるため、レア種の羽冠だけを見ても
   「来訪→観察→羽冠ドロップ」の**二重の確率**を通る。現状これに**下限(pity)がなく**、
   運が悪いと羽冠だけが長期間出ないケースがありうる。原則2は「罰しない」であって
   「運の分散をゼロにする」ではないが、Unmet Need対策としては手つかず。
2. **`app.py` の「使い方」タブが実装と不一致**(実害あり、要修正)。`tab_help`
   (`app.py:2774-2791`)は「羽根5%/種5%/小枝7%/木の実4%/羽冠2%、全26種、
   全コンプリートで5×26=130種」という**旧・5カテゴリ仕様**を説明しているが、実装
   (`mementos.py`)は**3カテゴリ(twig10%/feather5%/plume4%)、plumeはPLUME_BIRDS(9種)のみ**で、
   現在の総鳥種35種から算出すると全カタログは35+35+9=**79種**(130ではない)。
   ユーザーが説明文どおりの確率・総数を期待して「思ったより出ない/多すぎる」と感じるのは、
   運ではなく**表示の誤り**が原因になりうる。これは Unmet Need #1 と同じ症状(苛立ち)を
   別の原因(ドキュメント不整合)から生む。**開発部への修正依頼を推奨**(企画としては数値の
   出典元を `mementos.py` に一本化するよう明記)。

**提案(数値バランス、企画部の論点として)**:
- **familiarity(既存 `observed[bird]["count"]`)に応じた plume 確率の緩やかな底上げ**を提案する。
  ```
  plume_prob(bird) = 0.04 + min(0.06, 0.01 × floor(observed_count / 5))
  # 観察5回ごとに+1%、上限+6%(最大10%)。twig/featherは現状維持。
  ```
  根拠: `radio.py` の `_obs_to_depth()` や `flock.py` の `flock_size()` が既に
  「観察回数(馴染み)が増えるほど体験が豊かになる」という前例を持つ(§3-4, §4-3)。
  同じ考え方を memento に拡張するだけで、**新しい仕組みを増やさず既存データを再利用**できる。
  ナラティブは「よく会う鳥ほど、小さな違いに気づけるようになる」——実際のバードウォッチャーが
  経験を積むほど観察眼が上がる、という**観察者側の学び**の話であって、鳥の生態を歪める話では
  ないため原則4に抵触しない(GloBIベースの食物網・reason_textには一切手を触れない)。
  - **表示しない**: `HANDOFF §8`「進捗バー・警戒度数値の非表示」の精神を踏襲し、確率や
    「あと◯回で確定」のようなカウンターは出さない。上がったことも明示せず、
    「羽冠が出た」時にだけ「よく会っているおかげかもしれません」と一言添える程度に留める。
  - これは**下限を保証するものではなく、外れが続いた時の分散を穏やかに狭める**設計であり、
    「損をした」感を和らげる方向にのみ働く(上がる一方で下がらない)。
  - 数値バランスの最終値(0.01/0.06/floor(5)の係数)は企画部の暫定案であり、実データ
    (訪問頻度の分布)を見て開発部と調整する前提。

**5原則チェック**: 受動的(条件・行動不要、既存データの副産物)/罰しない(悪化方向の調整なし)/
声と癒しは常に無料(memento確率にすぎず声のアクセスとは無関係)/生態に誠実(食物網・reason_textは不変、
観察者スキルの話に限定)/かわいさ(表示は最小限、羽冠獲得時の一言のみ)。**CEO確認は不要**と判断
(HANDOFF §8の確定済み設計判断を覆さず、★★★核機能表にも触れない、`05_運用ルール.md` §1の
企画部承認ラインにも該当しない数値バランス調整)。ただし実装時は開発部の通常レビューを通すこと。

### 1-2. 収集完了後の頭打ち(ねこあつめ・旅かえる) — **手つかず。部長仮説を支持し、優先度を上げるべき**

**現状確認**: `CONCEPT_v2.md`拡張構想v2.1の「実装の当たり(小さく出す順)」は
①日次カウント(✅済)→②シミュ・ラボ→③おやつ→④収集→交換、の順。`01_企画部.md`
「次に企画で詰めること」も①シミュ・ラボ②おやつ③通知④数値バランスの順で、
「収集→交換」はどちらの優先度リストにも**上位に出てこない**。

**部長仮説「2は"収集→交換"ロードマップの重要度を上げるべき」を支持する。** 根拠:
- 現在の鳥種は35種(`HANDOFF §2-3`)。`daily.py`の「今日の一羽」と不在ループにより
  比較的早いペースで新規発見が進む設計であり、**図鑑(`discovered`, 有限集合)は
  遠くない将来にコンプリートしうる規模感**。コンプリート後に何もなければ、
  ねこあつめ・旅かえるが陥った「やることがなくなる」を再現するリスクは実在する。
- 一方で、**Torisには既にこの問題への"半分の答え"がある**ことも確認できた:
  「会った日数」(`HANDOFF §4-4`)は**上限のない蓄積指標**であり、図鑑(有限)がコンプリートした
  後も、同じ鳥に「また会う」たびに増え続ける。ねこあつめには存在しない**第二の軸**が
  既に実装されている。これは意図的か偶然か問わず、Unmet Need #2への構造的な備えとして機能しうる。
- ただし、**この第二の軸に"意味"を与える機能がまだ無い**。会った日数が増えても、
  節目(10/30/100日)の演出や見返り(前回提案書「案C」で提起済み)が未実装のままだと、
  「増えているだけで何も起きない」数字になり、頭打ち対策として機能しない。
  → **だからこそ「収集→交換」の優先度を上げるべき**、というのが本提案の結論。

**優先度引き上げの提案**: `CONCEPT_v2.md`拡張構想v2.1の「小さく出す順」を
①日次カウント(済)→**②収集→交換の第一歩(新規に繰り上げ)**→③シミュ・ラボ→④おやつ、
に組み替えることを提案する。理由: シミュ・ラボとおやつは「遊びの幅」を広げる施策であり
Unmet Need #2(頭打ち)への直接対処ではない。収集→交換は、**既に実装済みの無限軸(会った日数)に
意味を与える最短距離の一手**であり、新規種・新規アセット制作を伴わないため
コスト効率も最も良い。

**小さな最初の一歩(具体設計)**: 前回提案書「案C」をさらに小さく2段階に分割する。
- **第一歩(データ追加ゼロ)**: 節目(10/30/100日)到達時に、図鑑の該当鳥のカードに
  **静かなバッジ演出のみ**を表示する(「よく会う仲になりました」等、既存の `load_bird_days`
  データのみで完結)。文章コンテンツの新規執筆は不要なため、実装規模は前回提案書より小さい。
- **第二歩**: 各バイオームの代表種1〜2種(計5〜6種、京都/シドニー/シャーロットから)に限定して、
  節目到達時に**豆知識カード**(GloBI由来の食性・関係性、事実ベース+出典明記)を解錠する。
  対象種を絞ることで「コンテンツ制作コスト」対「頭打ち対策の効果」のバランスを取り、
  反応を見てから全種に拡張するかを判断する。
- 課金・外部提携(アフィリエイト等)は**この最初の一歩には含めない**。将来検討時に
  `05_運用ルール.md` §1「収益モデル・課金に関わる新規提案」としてCEO確認を別途取る。

**5原則チェック**: 前回提案書「案C」のチェックをそのまま踏襲する。
- 受動的: 既存の非連続・累計カウント(1日1回、途切れても損しない)に完全に乗る。
  「連続ログイン◯日」への変更は絶対にしない。
- 罰しない: 会った日数は撹乱で減らない(`HANDOFF §1-1-a`と同じ思想)。本提案は加算のみ。
- 声と癒しは無料: 豆知識カードの閲覧は無料。将来のアフィリエイトも「基本閲覧は無料、
  付加価値のみ収益化」の一線を守る。
- 生態に誠実: 豆知識は GloBI由来の事実または出典のある情報に限定。作り話の逸話は厳禁。
- かわいさ: バッジ・カードは鳥のドット絵と鳴き声を主役にし、単なる数値カウンターにしない。

**CEO確認の要否**: この優先度組み替えは`CONCEPT_v2.md`の★★★核機能表(機能の再定義)を
変更するものではなく、`05_運用ルール.md` §1の企画部承認ラインにも該当しない
(拡張構想v2.1内の実装順の並べ替えに留まる)。**CEO確認は不要**と判断するが、
正典(`CONCEPT_v2.md`)への反映は部長経由で行うため、その際に一度目を通していただくことを推奨する。

### 1-3. 続編での使い回し批判(ねこあつめ2) — **手つかずのリスク。将来の拡張時に注意が必要(部長仮説を支持)**

**現状確認**: `data.py` の `SPRITE_ALIASES`(680-694行)を見ると、**既に現行3バイオーム内で
複数種がドット絵を共有**している(例: `shirohara→american_robin`、`mozu→hiyodori`、
`aoji→kawarahiwa` など、確認できた範囲で12組)。つまり「見た目の使い回し」は
MVP時点から**構造として存在する**(容量・制作コストの都合による意図的な設計判断と推測される)。

一方で、**鳴き声(xeno-canto実録音)は種ごとに本物であり、これは"音楽の使い回し"には
原理的にならない**。ねこあつめ2への批判は「同じ絵・同じ"音楽"」という2点セットだったが、
Torisは声の面では常に構造的にクリアできる強みを持つ。

**リスクが顕在化する条件**: `HANDOFF §12`「未着手・次フェーズ候補」の1番目
「鳥種の拡充(〜50種へ)」や、京都/シドニー/シャーロット以外への新規バイオーム追加の際、
**スプライト使い回し比率がさらに高まると**、見た目の新規性の薄さが「使い回し」という
形で不満化するリスクがある(CEO依頼文の懸念と一致)。

**提案(ガイドライン、実装は開発部/アセット制作側マター)**:
1. 新規バイオーム・新規種を追加する際、**専用ドット絵の最低比率**を企画側の基準として
   設ける(例: 新規追加種のうち専用絵を用意する種の比率を事前に決めてから着手する)。
   具体的な比率は開発部のアセット制作コストと相談して決めるべきであり、本書では
   「無制限のエイリアス化に頼らない」という方針レベルの提案に留める。
2. エイリアスを使う場合も、`color`フィールド(既に全種が保持)による配色差別化を
   **必須の最低ライン**とする(現状も色は種ごとに異なるため、この基準は現状追認に近い。
   今後増えても崩さないことの明文化)。
3. **鳴き声が実物であることを差別化の前面に押し出す**(SNS部・ASO文言と連携)。
   「絵は近くても、声は本物」というメッセージは、見た目の使い回しへの批判の
   防波堤として機能しうる。これはSNS部の管轄なので、本書では方針提示のみ。

**5原則チェック**: 受動的/罰しない/声と癒しは無料 — 本項目はいずれも直接関与しない
(表現ガイドラインのため)。生態に誠実 — 見た目の共有は生態的な誤りを生まない
(学名・食性・鳴き声は種ごとに正確なまま)ため抵触しない。かわいさ最優先 —
**本項目が最も直接に関わる原則**。使い回しの許容ライン(比率・配色差別化)を
決めずに種数だけ増やすと、原則5(かわいさ最優先)の質が下がるリスクがある。

**CEO確認の要否**: 本項目はガイドライン提案であり、`HANDOFF §12`の「鳥種拡充」自体を
覆すものではない。「やらない/後回し」の確定項目にも触れない。**着手判断そのものは
今回不要**(まだ拡張の実行フェーズではない)が、実際に新規バイオーム追加に着手する
タイミングで、本ガイドラインを踏まえた予算・比率の判断としてCEO/開発部と改めて
すり合わせることを推奨する。

## 2. Wingspanの知見(「本物の生態的正しさ自体が快感になる」)の活用 — 機能案2つ

`01_企画部.md`(26行目)に既にある論点「関係のプレビュー(植える→来る鳥)を見せて学びを前に出す」を、
Wingspanの具体的な手法(各カードが実際の生態行動を"パワー"として明示し、それ自体が快感になる)を
参照して深掘りする。**新規論点の追加ではなく、既存論点の具体化**という位置づけ。

### 案1: 「植物カード」化 — 植える前に、生態的役割を"パワー"として見せる

**概要**: 「植える」タブで植物を選ぶ際、その植物が実際にどの`eats_plants`関係(`data.py`)を
持つかを、Wingspanのカードのような**短い一文の"役割"**として見せる。
例: 「サクラ — 花に虫が集まり、虫を狙う鳥を呼びやすくなります」。

**留意点(コア体験の順序を壊さないための線引き)**: `01_企画部.md`のコア体験順序
(離れる→時間が進む→発見=報酬→なぜ来たかがわかる→...)における「発見=報酬」を
先取りしてネタバレしないよう、**プレビューは「呼びやすくなる関係の"種類"」に留め、
「必ずこの鳥が来る」という確約や、到来のタイミングは明かさない**。これは既に
`01_企画部.md`が「関係のプレビュー」として承認済みのスコープと同じ線引きであり、
新しい踏み込みではない。

**5原則チェック**: 受動的(見るかどうかは任意、植える操作自体は既存)/罰しない(プレビューを
見なくても不利益なし)/声と癒しは無料(テキストのみ、声のアクセスに影響なし)/
**生態に誠実(最重要)**: 表示する一文は`eats_plants`/`eats_insects`という既存の実データ
から機械的に生成し、恣意的な逸話を足さない(`species_expand.py`冒頭のコメント
「"なぜ来たか"の恣意ロジックは作らない」を植物側にもそのまま適用する)/
かわいさ(テキストが増えすぎると図鑑的な無機質さに寄るリスクがあるため、
一文・アイコン程度に抑え、既存の色・ドット絵を主役に保つこと)。

### 案2: 「生態ログ」— 一度きりの到来理由テキストを、図鑑に蓄積する永続的な"学びの記録"にする

**概要**: `absence_loop.build_reason_text()`(59-85行)が生成する「なぜ来たか」の一文は、
現状**到来時に一度表示されて終わり**(`app.py`のバナー表示、揮発性)。これを図鑑の各鳥の
ページに、**これまでに記録された来訪理由を重複除去して蓄積表示**する形に変える。
例: 図鑑のシジュウカラのページに「サクラに惹かれて/毛虫を狙って」等、これまで観測された
関係の一覧が並ぶ。

**狙い**: Wingspanのカードが「一度読んだら終わり」ではなく**手元に残り、何度も読み返せる
生態的事実の資産**になっている点を参照する。Torisの`reason_text`は既に生態学的に正確な
情報(GloBI由来)を生成しているのに、現状は**流れて消える演出**にしか使われていない。
これを蓄積させるだけで、既存の計算パイプライン(`build_reason_text`)を変えずに
「遊ぶほど生態系がわかる」(`00_共通サマリ.md`一行目)という一行サマリーを、
図鑑という**すでにある場所**でより強く体現できる。

**5原則チェック**: 受動的(自動蓄積、追加操作不要)/**罰しない(重要な確認点)**:
`HANDOFF §1-1-a`「ラジオ=蓄積するコレクション、記録は減らない」と同じ思想で、
一度記録された来訪理由は撹乱で植物が失われても図鑑からは消さない(過去の記録として残す)/
声と癒しは無料(テキストのみ)/生態に誠実(既存`build_reason_text`の出力をそのまま使うため、
新しい恣意ロジックの追加は一切ない。これは案1よりも生態的誠実性のリスクが低い、
**最も安全な実装**と言える)/かわいさ(図鑑ページが文字だらけにならないよう、
一覧は短文・アイコン付きの3〜4行程度に抑える)。

**推奨順位**: 案2(生態ログ)→案1(植物カード)。案2は既存データの見せ方を変えるだけで
新しい生成ロジックが要らず、生態的誠実性のリスクも最小。案1は「事前に見せる」ことに
コア体験順序との整合の慎重な線引きが必要なため、案2で反応を見てから着手する順を推奨する。

**CEO確認の要否**: いずれも`CONCEPT_v2.md`の★★★核機能表を変更する提案ではなく、
既に企画部が保有する論点(「関係のプレビュー」「なぜ来たか=生態の可視化」)の具体化に
留まるため、**CEO確認は不要**と判断する。図鑑UIへの変更を伴うため、実装時は
開発部の通常の「壊さない検証」を経ること。
