# Streamlit Community Cloud 以外のホスティングルート比較・実行準備(2026-07-08・開発部)

## 背景・目的

`https://toriscollection-test202605.streamlit.app/` は依然として Streamlit Cloud 側の
認証画面(`https://share.streamlit.io/-/auth/app?...`)へ303リダイレクトされる状態が続いており
(`docs/team/proposals/2026-07-07_Phase1実行計画.md` §0-1 で確認済み、2026-07-08時点も未解決)、
Google Play進出(Capacitor版のWebViewが読み込むURL)・一般公開のどちらにとっても障害になっている。

本ドキュメントは、CEOの指示(「Streamlit Community Cloud以外のルートも作って実装したい」)を受け、
(a) 今の不具合から独立した公開URLを別ホストに用意すること、(b) 将来TWAを再検討する余地
(`assetlinks.json`のルート配信)を残すこと、の2点を目的にホスティング候補を比較し、
実行に必要な設定ファイルをリポジトリに用意したもの。

**本ドキュメント作成時点でこのツールから外部Webページへ直接アクセスする手段がなく、
下記の比較は主に一次情報(各サービスの公式ドキュメント)に基づく知識と、リポジトリの
実際の構成の確認に基づく。無料枠の条件・料金・CC要否は変更される可能性があるため、
CEOが実際にサインアップする際に必ず各サービスの最新のプライシングページで裏取りすること
(下記の表にその旨を明記している)。**

## 1. 前提として押さえておくべき技術的事実

`2026-07-07_Phase1実行計画.md` の技術検証(§2 B)により、**「未知のパス
(`/.well-known/assetlinks.json` 等)へのアクセスがindex.htmlにフォールバックする」挙動は
Streamlit本体のTornadoルーティング実装そのものに起因する**ことが確定している。
つまり、**Streamlit Cloudから別のホスティング先(Render/HF Spaces等)に移すだけでは、
assetlinks.jsonのルート配信問題は自動的には解決しない**(Streamlitプロセスを素で動かす限り、
どこにホストしても同じ制約を持つ)。

ただし、Streamlit Community Cloudと異なり、**Render・Fly.io・Google Cloud Runのような
「自分のコンテナ/プロセスを丸ごと動かせる」ホスティングでは、将来 Docker + 簡易リバース
プロキシ(nginx等)を前段に置き、`/.well-known/assetlinks.json` だけを静的ファイルとして
横取りして返す構成に変更する余地がある**(Streamlit Community Cloudにはその余地が
そもそも無い、という違い)。今回はこの構成までは作り込まない(スコープ外)が、
「将来TWAを再検討する余地を残す」という目的には、こうした汎用ホスティングを選ぶこと自体が
資する。

今回の最優先目的は (a) の「今すぐ独立して公開できるURLを用意する」ことなので、
以下の比較・準備はまずそこにフォーカスし、(b) は「選択肢を閉じない」という観点で評価に含めた。

## 3. 推奨: Render.com を本命、Hugging Face Spaces を次点とする

### なぜ Render.com か

1. **リポジトリ構成の変更が一切不要**。`render.yaml`の`rootDir: toris_collection`により、
   このモノレポ(リポジトリ直下に`README.md`・`android_app/`等が同居し、Streamlitアプリは
   `toris_collection/`サブディレクトリにある構成)をそのまま接続できる。
   Hugging Face SpacesはSpaceリポジトリの直下に`sdk`指定つきの`README.md`を置く前提が強く、
   このモノレポをそのまま自動同期する場合は既存のルート`README.md`(プロジェクト全体の顔となる
   ドキュメント)にHF専用のYAMLフロントマターを足す必要が生じ、GitHub上での見え方に
   影響しうる(§4で詳述)。この一手間・副作用を避けられる点でRenderを優先した。
2. 無料のWebサービスとして公式にPythonアプリのデプロイに対応しており、追加のDockerfileも不要。
3. `git push`のたびに自動再デプロイされるため、「サーバー運用の負荷を増やしたくない」という
   CEOの意向(`ROADMAP_GOOGLE_PLAY.md`§0参照)に合致し、継続的な手間がほぼ発生しない。
4. 将来Docker運用に切り替える余地があり、(b)の「TWA再検討の余地を残す」も満たせる。

### 弱点として申し送りしておくこと

- 無料プランはアイドル時にスリープし、次のアクセスでコールドスタート(数十秒〜1分)が発生する。
  受動的な癒しアプリという性質上、致命的な体験劣化ではないと判断するが、体感を確認してから
  判断してほしい。
- 無料プランの条件(月間の実行時間上限等)は変更されうるため、実際にサインアップする際に
  最新のプライシングページで確認すること。

### Hugging Face Spaces を選ぶ場合(次点・Plan B)

Render側で問題が出た場合や、CEOがHFの方を好む場合のための代替案。手順の要点のみ記す
(実装ファイルは今回は用意していない。理由: 上記の通りモノレポ構成との相性で一手間かかり、
かつ選ばれるかどうか未確定の段階でルートの`README.md`に手を入れるのは影響が大きいため)。

1. HF上で新規Space作成、SDKに`streamlit`を選択。
2. Spaceは独立したgitリポジトリなので、以下のいずれかで中身を持っていく:
   - `git subtree push --prefix=toris_collection <space-remote> main`
     (`toris_collection/`配下だけをSpaceのルートとして送る。この場合、送る前に
     `toris_collection/README.md`の先頭にHF指定のYAMLフロントマター
     `---\nsdk: streamlit\napp_file: app.py\n---`を足す必要があり、この変更は
     GitHub上でも同じファイルに反映されるため、GitHub側の見え方にも影響する)
   - またはSpaceの管理画面から手動でファイル一式をアップロード(git操作不要だが、
     以後の更新のたびに手動再アップロードが必要になり「ほぼ放置」の方針からは外れる)
3. `XC_API_KEY`・`FREESOUND_API_KEY`をSpaceの「Repository secrets」に設定
   (HF Spacesのsecretsは実行時に環境変数として渡るため、`xc_client.py`/`freesound_client.py`の
   既存の環境変数解決ロジックをそのまま利用できる。コード変更不要)。

## 4. 今回リポジトリに用意したもの(実行はしていない)

- **`render.yaml`**(リポジトリ直下): Render Blueprint設定。`rootDir: toris_collection`で
  既存の`app.py`をそのままデプロイできるようにした。ファイルを置いただけでは何もデプロイ
  されない。CEOがRenderにサインアップし、このリポジトリを「Blueprint」として接続して
  初めてサービスが作成される。設定の詳細・CEOが行う手順はファイル冒頭のコメントに記載済み。
- 既存の`toris_collection/app.py`等のアプリコードは一切変更していない(制約の遵守を確認済み)。

## 5. CEOに実際に行っていただく必要がある操作(まとめ)

開発部からは実行できない(外部サービスのアカウント作成・接続操作のため):

1. https://render.com にサインアップ(GitHub連携推奨)。サインアップ画面でクレジットカード
   登録を求められるかを実地で確認していただきたい(本ドキュメントでは断定できていない)。
2. Render ダッシュボードで「New +」→「Blueprint」→ このリポジトリ(`AT-KB/toriscollection`)
   を接続。`render.yaml`が自動検出され、サービス作成が始まる。
3. 必要であれば環境変数(`XC_API_KEY`・`FREESOUND_API_KEY`)をRenderのダッシュボードで設定。
4. デプロイ完了後に発行されるURL(`https://<サービス名>.onrender.com`)にアクセスし、
   実際に動作することを確認。
5. 動作確認が取れたら、`android_app/capacitor.config.json`の`server.url`を新URLに更新するかを
   判断(現在は壊れているStreamlit CloudのURLを指しているため、Capacitor版のAndroidアプリを
   実際にビルド・配布する前には更新が必要になる。**この変更は「本番デプロイに関わる設定変更」
   として、実施前に改めてCEO確認を取ることを推奨する**)。

## 6. 承認・確認が必要な事項

- `05_運用ルール.md`§1の基準に照らし、**Renderへの実際のサインアップ・接続・公開("本番デプロイ")
  はCEO自身の操作が前提**であり、開発部が代行するものではない(アカウント作成が必要なため
  技術的にも代行不可)。
- `android_app/capacitor.config.json`の`server.url`変更は、Androidアプリがユーザーに見える形で
  読み込むURLを変える恒久的な設定変更のため、実施前にCEO確認を取る(今回は変更していない)。
- 無料プランの条件(CC要否・料金発生条件)は必ず最新のサービス側ページで裏取りしてから
  登録すること。
