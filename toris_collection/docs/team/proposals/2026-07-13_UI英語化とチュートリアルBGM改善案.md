## 1. UI英語化の方針

### 1-1. まず確認したい点(意図が明言されていない)

`ROADMAP_GOOGLE_PLAY.md`・`DIRECTION_BUSINESS_MODEL.md`・`docs/team/04_社長.md` の
いずれにも、**英語化の動機となる海外展開・グローバル配信の方針は明記されていない**。
Google Play自体は日本語のみのアプリでも世界中に配信でき、英語化は技術的な提出要件ではない
(現に多くの日本語専用アプリがPlayで公開・DLされている)。つまり「英語化しないとPlayに出せない」
という制約は存在しない。

一方で `02_SNS部.md` のASOキーワード・発信フォーマットは現状すべて日本語であり、
これは「まず日本語圏での認知」を前提にした設計になっている。この前提とUI英語化の関係が
今のところ整理されていない。

**企画部としてCEOに確認したい最重要の論点**: UI英語化は、
(a) 海外(英語圏)ユーザーの獲得を狙った展開の一歩なのか、
(b) Google Playのグローバル配信を見据えた将来投資(今すぐ海外広告はしないが土台だけ作る)なのか、
(c) それ以外の理由(開発体験・個人の言語選好など)なのか。
この意図によって、下記1-2の「完全置換 vs i18n」のどちらが妥当かが大きく変わるため、
**技術判断の前に意図の確認を推奨する**。

### 1-2. 完全置換 と i18n(両方持たせる)の比較

| | A. 完全置換(日本語UIを英語に置き換え、日本語版は無くす) | B. i18n導入(言語切り替えで両方持たせる) |
|---|---|---|
| 実装コスト | 低い(既存の文字列を差し替えるだけ) | 高い(文字列辞書化・切替UI・全箇所の呼び出し変更。Streamlit上でのi18n実装の技術的難易度・保守性は**開発部の判断が必要**) |
| 保守コスト(今後) | 低い(単一言語のまま) | 継続的に高い(新機能を作るたび2言語分の文言を用意・同期する必要がある) |
| 既存(日本語)ユーザー体験 | 失われる。現在のテスター(`tester_01`等)は日本語話者と推測され、影響が出る | 維持できる。ユーザーは言語を選べる |
| SNS部の発信・ASOとの整合 | **不整合が生じる**。`02_SNS部.md` の日本語ASOキーワード・発信フォーマットを英語圏向けに作り直す必要が生じ、SNS部側の追加工数が発生する(SNS部との協議が別途必要) | 整合を保ちやすい(日本語UI利用者には従来どおりの体験を提供したまま、英語圏向け発信だけ別途検討できる) |
| 将来のスコープ拡張(鳥/植物名・アプリ名の英語化) | 一貫性が保ちやすい(既にUIが英語なので、名前表記を英語化する際の違和感が少ない) | 一貫性の管理コストが上がる(UIは英語/日本語両方あるのに、名前だけどちらかに寄せる、といった中途半端な状態が生まれやすい) |
| 交渉不能の原則との関係 | 直接抵触する原則はない(下記1-4参照) | 同左 |

### 1-3. スコープの曖昧さ(確認が必要)

CEOの依頼文では「UIラベル・ボタン文言(タブ名・ボタン・メニューなど画面上の操作系テキスト)」
が対象、と整理されている。しかし実装を見ると、Streamlit UIの文言は「操作系(ボタン・タブ名)」
と「説明・案内文(help/info/caption等の地の文)」がかなり混在しており、後者の分量が非常に多い
(例: `tutorial.py` の案内文、`radio.py` の各種 `help=`、`app.py` の `st.info`/`st.caption`
群、生態ログのナレーション文など)。「操作系テキストのみ英語化」という狭いスコープで進めると、
**ボタンは英語・説明文は日本語という混在状態になり、かえって分かりにくくなる懸念がある。**
このスコープ(ボタン・タブ名だけか、案内文・ヘルプテキストも含むか)を先に確定してから
着手範囲を見積もることを推奨する。

### 1-4. 交渉不能の原則との整合

| 原則 | 判定 | 備考 |
|---|---|---|
| 受動的 | 影響なし | 表示言語は体験のテンポやせかし方に関与しない |
| 罰しない | 影響なし | 同上 |
| 鳥の声と癒しは常に無料 | 影響なし | 課金構造に触れない |
| 生態に誠実 | **スコープ外(今回は影響なし)** | 鳥/植物の名前表記はCEOが明言どおり対象外。ただし将来スコープが広がり学名/和名/英名の表記を変える場合は、GloBIベースの関係性の記述(理由文・生態ログ)が英訳時に意味がずれないよう、企画部が改めて監修する必要がある(軽く留意点として記録) |
| かわいさ最優先 | **要留意(致命的ではない)** | 現在のUIはやわらかい日本語の言い回し(例:「あとは待つだけ」「そっと」等)で癒しのトーンを作っている箇所が多い。直訳的な英語化だとこの"やわらかさ"のトーンが失われるリスクがある。単なる翻訳ではなく、英語圏の癒し系アプリのトーンに合わせたコピーライティングとして扱うべき(機械的な逐語訳を避ける)、という点をCEOに共有しておきたい |

### 1-5. 企画部としての推奨(判断材料の提示)

1. **意図の確認が最優先**(1-1)。海外ユーザー獲得が目的なら、UI英語化だけでなく
   ASO・SNS発信・音源のライセンス構成(既存は日本語圏の鳥中心)まで含めた展開計画が必要になり、
   企画部・SNS部・開発部の三部署にまたがる大きめの意思決定になる。単なる「言語の変更」では
   終わらない可能性がある。
2. 「今のテスター・初期ユーザーが日本語話者中心」という現状(`docs/team/04_社長.md` の
   3〜6ヶ月の見立てが日本語圏での継続率観測を前提にしている)を踏まえると、
   **完全置換は既存の観測計画と衝突するリスクがある**。i18nで両方持たせるほうが安全に見えるが、
   実装・保守コストの増加は無視できない(Streamlit上のi18nの技術的難易度は開発部確認が必須)。
3. 上記を踏まえ、企画部としては「まず意図を確認し、(a)海外展開が目的なら i18n 導入を
   開発部に技術検討させたうえで段階導入、(b)将来投資や個人の選好が理由なら、今は焦って
   全面着手せず、スコープ(1-3)を先に確定してから小さく試す」という進め方を判断材料として提示する。
   **どちらの方向で進めるかはCEO確認必須**(05_運用ルール §1「ユーザーに見える形で公開される」
   に該当する変更のため)。

## 2. チュートリアルの改善案

### 2-1. 現状整理(確認済みの事実)

- 全3ステップ: 0=土地を選ぶ、1=植物を植える、2=最終案内。
- ステップ1→2は既に「実際に1つ植えたら自動で次に進む」(`tutorial.resolve_step`)。
- ステップ0は `app.py` L2762-2812 の土地選択UI(`st.radio` + 変更時のみ現れる
  「✓ ◯◯に移る」確定ボタン)と**連動していない**。チュートリアル側の「次へ」ボタンは
  常にクリックするだけで進める、独立した別ボタン(L2203)。
  - **見つかった副次的な事実**: 現状は「土地の候補(`biome_candidate`)を切り替えても、
    土地選択UI側の『✓ ◯◯に移る』を押さない限り実際の土地(`st.session_state.biome`)は
    変わらない」。この状態でチュートリアルの「次へ」を押すと、選んだつもりの候補が
    反映されないまま次ステップに進む。これは「実行したかどうかのチェックが無い」ことの
    延長で起きる**UXの分かりにくさ**であり、CEOのフィードバックの背景の一つと考えられる。
- ステップ2の文言「準備はできました。あとは待つだけ」+ タイトル「🎉 ステップ3/3」が、
  完了を祝う演出(🎉)になっており、「植えるとすぐ終わる」ように読める。実際は
  コア体験(`01_企画部.md`:離れる→時間が進む→戻って発見)の**入口**でしかない。

### 2-2. ステップ0を「実際に選んだ」ことをもって進める案

**大前提(交渉不能の原則1・2)**: どんな設計でも、①いつでも即座にスキップできる導線を
残す、②選ばなくても・スキップしても一切のペナルティ(鳥が減る等)を発生させない、
の2点は必須。以下はこの前提の範囲内での設計案。

企画部としては、リスクの異なる2案を用意した。

#### 案A(推奨・低リスク): ボタンの意味づけを変えるだけ(ブロックなし)

- チュートリアルの「次へ →」という**汎用のページ送りラベル**を、ステップ0限定で
  「🏞️ この土地でいく →」のような**行為そのものを言い表すラベル**に変える。
- 同時に、このボタンのクリック時に、現在 `biome_candidate`(候補)と
  `st.session_state.biome`(実際の土地)が異なっていれば、既存の「✓ ◯◯に移る」と
  同じ処理(土地の切替・植生リセット等)を**このボタン一つで実行**するようにする
  (=候補を選んでいた場合はここで初めて反映される。前述2-1の分かりにくさの解消にもなる)。
- 「次へ」を押す=「今見えている土地で庭を始める」という**実質的な行為**になるため、
  技術的なブロック(押せない状態)を一切作らずに「実行して進む」感覚を作れる。
- スキップボタンは現状のまま常に併存させる。
- 実装コストが低く、原則1・2に一切触れないため、**最も安全な第一歩**として推奨する。

#### 案B(CEOの提案により近い・要確認): 「次へ」を軽くゲートする

- 案Aに加え、`tutorial_land_touched`(仮称)という session_state フラグを新設。
  - 土地選択UI(`biome_candidate`)を一度でも操作した、または案Aの確定ボタンを
    一度クリックした時点で `True` にする。
- ステップ0の「次へ」ボタンは、`tutorial_land_touched` が `False` のうちは
  **淡いトーンで disabled 表示**にし、代わりに「🏞️ 庭の様子タブで土地を確認してみましょう」
  という誘導文を出す。
  - ただし**「チュートリアルをスキップ」ボタンは常にフル活性・同じ視覚的重みで隣に表示**し
    続ける(=いつでも1クリックで全体を終了できる安全弁は変えない)。
- **この案は「押せないボタン」を一時的に作る**ため、原則1「受動的である(急かす設計を
  入れない)」の解釈にかかわるグレーゾーンである。スタミナ制・時短課金・デイリー強制のような
  「待たせる/課金させる」類の急かしとは性質が異なり、**ペナルティも時間消費もない
  ワンクリックの行動を促すだけ**ではあるが、「ボタンが一時的に押せなくなる」UI自体は
  Toris Collection にこれまで存在しない種類の制約であるため、**導入するかどうかはCEO確認を
  推奨する**(05_運用ルール §1「交渉不能の原則に触れる可能性があれば事前承認」に該当)。

**企画部の推奨**: まず案Aを実装し、様子を見てから、まだ「実行感」が足りないと感じる場合に
案Bを検討する、という段階導入を推奨する。

### 2-3. ステップ2の文言修正案

現状:
```
title: "🎉 ステップ 3/3: あとは待つだけ"
body:  "準備はできました。アプリを閉じていても、庭の生態系は時間とともに動き続けます。
        次に開いたとき、「🎙 ラジオ」タブで鳴き声を、「📖 図鑑」タブで出会った鳥を
        確かめてみてください。"
next_label: "はじめる ✓"
```

**問題点**: 🎉(完了の演出)+「あとは待つだけ」という言い回しが、「もうゴールした」
という誤解を生む。実際にはここが**コア体験(離れる→時間が進む→発見)の起点**であり、
「今はまだ何も起きていない」ことこそ伝えるべき情報。

**修正案**:
```
title: "⏳ ステップ 3/3: ここからが本番です"
body:  "今はまだ、庭は静かなままです。ここでアプリを閉じて、少し時間をおいて
        (数時間後や翌日など)また開いてみてください。あなたが離れている間に
        生態系が動き、鳥が来ます。戻ってきたら「🎙 ラジオ」タブで鳴き声を、
        「📖 図鑑」タブで出会った鳥を確かめられます。"
next_label: "わかった、庭を育てにいく →"
```

- 「今はまだ〜」で完了ではないことを明示。
- 「離れる→時間が進む→戻って発見」という `01_企画部.md` のコア体験の順序をそのまま
  文言に落とし込み、何が起きる/起きないのかの期待値を正しく揃える。
- 強制的な煽り文句(「〜しないと」)は使わない。既存テスト
  `tests/test_tutorial.py::test_step_content_final_step_has_finish_label_not_skip` が
  `"しないと" not in content["body"]` を明示的に検証しており、この非強制の方針は
  今回の修正案でも維持している(実装時、この既存テストとの整合を保つこと)。
- 🎉(完了の華やかな演出)は絵文字を⏳(経過待ち)に変更し、視覚的にも「まだ終わっていない」
  トーンに寄せる。

### 2-4. 交渉不能の原則との整合(チュートリアル全体)

| 原則 | 判定 | 備考 |
|---|---|---|
| 受動的 | 案A/文言修正=Yes、案B=**要確認**(2-2参照) | コア体験のテンポ(離れる→時間が進む)自体には触れない。オンボーディングの一時的な導線のみが対象 |
| 罰しない | Yes | どの案でもスキップ時・未選択時のペナルティ(鳥減少等)は一切発生しない |
| 声と癒しは無料 | 影響なし | 課金・アクセス制限に無関係 |
| 生態に誠実 | 影響なし | 文言・UI導線のみの変更 |
| かわいさ最優先 | Yes | 演出トーンをむしろ「完了の断定」から「静かな経過待ち」に寄せ、癒し系の世界観に近づける方向の修正 |

## 3. BGMボタンの体感改善

### 3-1. 現状整理(確認済みの事実)

- `radio.py` のBGMトグルはパラメータ微調整方式(トラック切替ではない)。ON時:
  - 鳥の発声頻度・音量を抑制(`RA_TARGET_ACTIVE` 1.2→0.4、`RA_REST_MIN/MAX_S` を延長、
    `gain.gain.value` を ×0.75 等)。
  - 環境音(パッド)の音量を `padBus.gain` 0.06→0.11 に引き上げ(約1.8倍)。
  - パラメータ変更はすべて `setTargetAtTime(..., t, 5.0)` という**5秒の緩やかな時定数**で
    適用されており、切替の瞬間に「変わった」と気づきにくい可能性がある。
- 本番ログで「[freesound] API key未設定、シンセ音にフォールバック」を確認済み。
  `freesound_client.get_ambient_path()` が `None` を返すため `HAS_AMBIENT=false` となり、
  ヒーリングBGMの環境音は実録音ではなく**シンセ合成の和音パッド+微弱な風音**になっている
  可能性が高い(`radio.py` L669-701 の `buildAmbient()` フォールバック分岐)。
- `DIRECTION_BUSINESS_MODEL.md` §10 に「ヒーリングBGMモード: 環境音(パッド)が主役になり、
  鳥は"ときどき1羽だけ短く"鳴く」という**既に確定している設計意図**があり、今回の改善案は
  この意図を強化する方向であって、新規に何かを付け足すものではない。

### 3-2. 体感を強めるための方向性(体験設計としての方針)

1. **ON/OFFの音量差そのものを広げる**: 現在の約1.8倍(0.06→0.11)という差は、5秒かけて
   じわじわ変化するため聴感上は「気づいたら変わっていた」程度に留まりやすい。
   「BGM ON時は、パッド(環境音)が鳥の声と少なくとも同等以上に前景として聞こえる」ことを
   体験設計上の目標値として置き、ON時のパッド音量をさらに引き上げる方向を提案する
   (具体的な dB/gain 値は開発部の音響調整領域だが、「ONにした瞬間、誰でも違いに気づける」
   ことをゴールラインとして共有したい)。
2. **切替の瞬間に気づかせる演出を足す**: 現状の `setTargetAtTime(..., t, 5.0)` は
   定常状態の心地よい"揺らぎ"には向くが、**トグル操作そのものへのフィードバックとしては
   遅すぎる**可能性がある。トグルを押した瞬間だけ、もう少し速い時定数(例: 1.5〜2秒程度)で
   パッド音量を立ち上げる/下げるようにし、「切り替えた」という行為への応答をはっきりさせる
   ことを提案する(定常時の揺らぎは現状の緩やかさのままでよい)。
3. **UI側の即時フィードバックを足す**: トグル操作の瞬間に、`st.toast()` 等で
   「🎧 ヒーリングBGM: 鳥の声が少なくなり、環境音がやわらかく主役になります」のような
   一言を一瞬表示する(既に `help=` に同内容の説明はあるが、hoverしないと見えない)。
   音そのものの変化に気づきにくい状況でも、「今何が起きたか」を言葉で補強できる、
   低リスクな改善。
4. **フォールバック時(シンセ合成)でも最低限の"環境音らしさ"を担保する**: Freesoundの
   実録音が無い状態でも、ヒーリングBGMの核である「環境音が主役になる」という約束は
   常に成立していなければならない(3本の柱「③癒し」は常に無料・常に機能する必要がある)。
   シンセパッド自体の質感(和音の厚み・風音の存在感)を、ON時に限り気持ち強めるのも
   選択肢の一つ(具体の音響パラメータは開発部判断)。

### 3-3. 開発部への確認依頼(企画部からは直接確認できない事項)

- **本番環境(Streamlit Cloud secrets)に `freesound_api_key` が実際に設定されているか**
  を開発部に確認してほしい。未設定であれば、CEOが体感した「効果が薄い」という感想の
  相当部分は「パラメータ微調整が弱い」以前に「そもそも実録音の環境音が鳴っていない
  (常にシンセ合成)」ことが主因である可能性が高い。この場合、まずAPIキー設定の要否
  (無料登録で取得可能、`freesound_client.py` 冒頭コメント参照)をCEOに確認し、
  設定できるなら設定した上で改めて体感を評価するのが優先度として高いと考える。
- 上記3-2の具体的な数値(gain値・時定数)の調整は開発部の音響設計領域。企画部としては
  「ONにした瞬間に誰でも違いに気づけること」「フォールバック時でも環境音が主役だと
  感じられること」を体験目標として渡す。

### 3-4. 交渉不能の原則との整合

| 原則 | 判定 | 備考 |
|---|---|---|
| 受動的 | Yes | BGMはユーザーが任意でON/OFFする機能。急かす要素は追加しない |
| 罰しない | Yes | ON/OFFどちらでもペナルティなし |
| 鳥の声と癒しは常に無料 | Yes、むしろ強化 | BGM(=癒しの3本目の柱)は元々無料機能。今回の改善は「無料の癒し体験の質」を上げる方向であり、課金要素は一切関与しない |
| 生態に誠実 | 影響なし | 音響パラメータ・UI通知のみの変更。生態ロジックには触れない |
| かわいさ最優先 | Yes | 「環境音が主役になる」という既存の癒し演出意図(`DIRECTION_BUSINESS_MODEL.md`§10)を強化する方向であり、既存トーンからの逸脱はない |
