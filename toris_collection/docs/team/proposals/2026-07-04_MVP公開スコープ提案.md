作成: 企画部 / 2026-07-04
位置づけ: 提案書。正典(`docs/team/01_企画部.md` 等)への反映は部長がCEOの確認を経て行う。

依頼の要約(PJ管理部長経由・CEO発):

セキュリティ負荷を軽減させたい。MVPでいい」

`app.py`(`render_login_screen` / `tab_community` / セッションキー周り)。

## 1. 一次情報の確認結果(要約)

- **`HANDOFF_2026-06.md` §8「確定済みの設計判断」** に
  「Google Sheets 永続化 | Streamlit Cloud のステートレス制約を回避」と明記されている。
  これは**確定済み判断**であり、覆す場合は `05_運用ルール.md` §1 によりCEO確認が必須。
  ただし、この判断が採用された**理由**は「Streamlit Cloud はサーバーが再起動するとメモリが
  消える(ステートレス)ので、何らかの外部永続化が要る」という**技術的制約の回避**であり、
  今回CEOが挙げている動機(「セキュリティ負荷の軽減」)とは**別の問題意識**である。
  つまり「覆す」というより「同じ問題(データをどこかに永続化する必要がある)に対して、
  Sheetsという解と、ローカル保存という解の、どちらを採るかの再選択」と整理できる。
  それでも正典を明示的に上書きする以上、**CEO確認は必須**として扱う。

- **`community.py` / `HANDOFF_2026-06.md` §4-6「みんなの庭」** は、複数テスターの
  `collection` シートを横断集計して「どの土地に何が来ているか」の匿名地図を作る機能。
  **構造的にサーバー側の集約データを前提とする**(1台のローカルデバイスの情報だけでは
  他人の庭は原理的に見えない)。`CONCEPT_v2.md` の機能表では**★☆☆(最低優先度)**、
  かつ3本の柱にも入っていない。`HANDOFF_2026-06.md` 自身が「基盤移行なしの実験」「点かなければ
  1タブ消すだけで撤去できる」と明記しており、**そもそも可逆な実験と位置づけられている**。

- **`app.py` の `render_login_screen()`**(443行目付近)は、Google Sheets の `testers` シートに
  事前登録された名前をプルダウンで選ぶ**クローズドテスト専用のログイン**。公開MVP(不特定多数が
  アクセスする想定)では、そもそも「使う人全員を事前にスプレッドシートへ登録する」運用は成立しない。
  つまり**ローカル保存に転換するかどうかに関わらず、公開時点でこのログイン方式は置き換えが必要**
  になる、という点を確認しておく。

- **`ARCHITECTURE_PLAN_2026-06.md` §5「Storage インターフェース」** は既に
  「画面・コアロジックは `Storage` という約束にだけ話しかけ、裏側(Sheets/SQL/Firebase)は
  差し替え自由」という設計を描いている。同文書が想定する裏側は「Sheets→もっと大きなサーバーDB」
  という方向だが、**「サーバー上の何か」から「デバイス上の何か」への差し替え**も、同じ
  `Storage` という約束の型には収まりうる(=既存の設計思想と方向として矛盾しない)。
  これは実装の型の話であり、企画としては「無理な逆行ではない」ことの確認に留める。

- **`daily.py`(今日の庭)** は `daily_seed(today)` という**日付のみに基づく決定的関数**で
  「今日の一羽」を決めており、サーバーとの同期なしに**全ユーザー共通の"今日"**が成立する
  設計になっている(HANDOFF §4-4)。ローカル保存に転換しても、この「全員に共通の今日」という
  性質は無傷で残る。

## 2. 核心の緊張関係の判定

「基本ローカルデバイス保存」を採ると、**ユーザー間で情報を持ち寄る機能(みんなの庭)は
構造的に成立しなくなる**。これは避けようのない事実であり、曖昧にせず結論を出す。

**結論: MVP公開時点では「みんなの庭」タブを非表示にする。**

理由:
1. `CONCEPT_v2.md` の優先度表で★☆☆(最低)。3本の柱(バイオーム構築/声を聴く/癒し)にも
   コア体験の順序(離れる→時間が進む→発見→図鑑・会った日数に積もる)にも含まれない。
   外しても**コア体験は一切損なわれない**。
2. `HANDOFF_2026-06.md` §4-6 が自ら「点かなければ1タブ消すだけで撤去できる」実験と
   位置づけている。つまり**この機能自体が「反応を見て消してもよい」前提で作られている**。
   今回はまだ公開実験の反応データがない段階なので、「消す」でなく「後回し(非表示)」とし、
   将来サーバー側の軽い仕組み(匿名の書き込み専用集計APIなど)を足すタイミングで再検討する
   、という扱いにする。コードは削除しない。
3. 代替として「ローカル保存のまま無理にみんなの庭を残す」ことも検討したが、それは
   結局サーバー側に何らかの匿名集計ストアを必要とし、「基本ローカル保存でセキュリティ負荷を
   下げる」というCEOの目的そのものと矛盾する。中途半端に両立させるより、**方針を一本化する
   ほうが誠実**と判断した。

## 3. MVP公開スコープの提案

### 3-1. そのまま残す(ローカル保存で自己完結する・コア体験そのもの)

| 機能 | 根拠 |
|---|---|
| バイオーム構築(植える・関係を組む) | 完全に1デバイス内で完結するデータ(`planted`) |
| 儀式(ritual.py・近距離観察) | 同上(`observed`) |
| ラジオ・ヒーリングBGM | 同上。3本の柱の中核、CEO動機と無関係に死守すべき部分 |
| 図鑑・落とし物(mementos) | 同上(`discovered`) |
| 会った日数(1日1回・日次カウント) | 同上。訪問ログの「異なる暦日」集計はローカルでも同様に計算可能 |
| あしあと(訪問カレンダー) | 同上 |
| 撹乱・遷移(disturbance.py) | 同上(`planted` の変化に付随) |
| 群れ(flock.py) | 同上(`observed` の count に付随) |
| 今日の庭(daily.py) | **日付のみで決まる純関数**。サーバー同期なしでも「全員共通の今日」が成立(§1参照) |
| ネットワーク図(食物網・tab_network) | 植えた植物ベース。1デバイス内で完結 |

これらは**現状のロジック(engine.py 等)を一切変えず、データの置き場所だけを変える**話であり、
コア体験・交渉不能の原則には触れない変更として整理する。

### 3-2. ログイン方式の変更(提案)

現行の「テスター選択プルダウン」(Sheetsの`testers`シート参照)を、**デバイスローカルの匿名識別子
(初回アクセス時に自動生成・ユーザー操作不要)**に置き換える。

- ユーザーは名前を選ぶ・登録する操作を一切しない。開けばすぐ庭が見える。
- 個人を特定する情報(名前・メール等)をサーバーに置かないため、CEOの動機
  (セキュリティ負荷軽減)に直接寄与する。
- **副次効果として受動性が改善する**: ログイン画面という「開いてすぐ本編に入れない」障壁が
  消える。`01_企画部.md` の「判断に迷ったら」の第一問(受動的か)にむしろプラスに働く。

### 3-3. 後回し・簡略化する機能(MVPには含めない)

| 機能 | 状態 | 後回しの理由 |
|---|---|---|
| みんなの庭(集合アトラス) | 実装済み・**MVPでは非表示** | §2 参照。構造的にサーバー集約が必要なため |
| シミュ・ラボ | 未実装 | 既存の優先順位どおり変更なし。今回のMVPスコープ外 |
| おやつ(餌台)本編UI配線 | ロジック実装済・UI未配線 | 既存の優先順位どおり変更なし。今回のMVPスコープ外 |
| 収集→交換(デジタル特典) | 未実装 | 同上 |
| 複数端末間の引き継ぎ・同期 | — | ローカル保存の裏返しの弱点。3-4で扱う |

### 3-4. データ喪失リスクへの手当て(要検討・企画からの推奨)

ローカル保存にすると、**ブラウザのキャッシュ消去・機種変更・別ブラウザでのアクセスなどで
図鑑・会った日数が失われうる**。これはゲーム内の行動(撹乱・不在等)による損失ではないが、
ユーザー体験としては「ある日突然すべて消える」という点で、交渉不能の原則2(罰しない)の
"精神"に触れうるリスクだと考える。

企画としての推奨(実装の要否・時期はCEO判断):
- 最低限、**「この端末にのみ保存されます。ブラウザデータを消すと図鑑が失われる場合があります」**
  という説明を初回起動時に一言添える(誠実な期待値調整。原則2は「ゲーム内で罰しない」が主眼
  であり、技術的制約の告知自体は罰ではないが、黙って失わせるのは避けたい)。
- 可能であれば軽量な**エクスポート/インポート**(進行状況を1つの文字列・ファイルとして
  書き出し・読み込みできる機能)を任意機能として用意する。強制はしない・目立たせすぎない
  (「バックアップ」的な扱いに留め、収集要素の主役にはしない)。
- これは**MVPの必須要件にはしない**(まず公開して反応を見る、というCEOの意図を優先する)。
  ただし「後回しにした」という事実は明示しておき、需要が見えた時点で優先度を上げられるようにする。

## 4. 交渉不能の5原則チェック

### (a) ストレージ方式の変更(Sheets→ローカル)そのもの
1. **受動的か**: Yes。永続化の場所の変更であり、急かす要素は増減しない。ログイン画面撤去は
   むしろ受動性を改善する(§3-2)。
2. **罰しないか**: 条件付きでYes。ゲーム内の行動で罰するわけではないが、**技術的な理由で
   進行が消えうるリスク**が新たに生じる。§3-4の期待値調整・任意のバックアップ手段で
   実質的な"理不尽さ"を薄める必要がある。**手当てなしで放置すると原則2の精神に反する
   おそれがある**ため、最低限の告知は必須要件として扱うことを推奨する。
3. **鳥の声と癒しは常に無料か**: Yes。ストレージ方式は課金・アクセス制御と無関係。むしろ
   サーバー側の個人データを持たない設計は「無料でずっと使える」という信頼にプラスに働く。
4. **生態に誠実か**: Yes。GloBIベースの関係・確率モデル(`engine.py` 等)には一切手を触れない。
5. **かわいさ最優先か**: Yes。UIの見た目・演出には影響しない変更。むしろログイン画面の撤去で
   「開いてすぐかわいい庭に会える」までの導線が短くなる。

**結論**: クリア可能。ただし「罰しないか」は無条件のYesではなく、§3-4の期待値調整を
**必須の付帯条件**とする。

### (b) 「みんなの庭」の非表示化(MVP版)
1. **受動的か**: Yes(影響なし。むしろタブが減り選択の負荷が下がる)。
2. **罰しないか**: Yes(既存の図鑑・会った日数など罰しない設計には無関係)。
3. **鳥の声と癒しは常に無料か**: Yes(無関係)。
4. **生態に誠実か**: Yes(無関係)。
5. **かわいさ最優先か**: Yes(むしろタブ構成がシンプルになり、コア体験に集中しやすくなる)。

**結論**: 5原則には抵触しない。ただし「実装済みの機能を公開版で見せない」という**スコープ判断**
そのものは、`05_運用ルール.md` §1 の「ユーザーに見える形で公開される」に該当するため、
交渉不能の原則の問題ではなく**運用ルール上の事前承認事項**としてCEO確認を求める(§5参照)。

## 5. CEO確認が必要な事項(`05_運用ルール.md` §1 に照らして)

1. **`HANDOFF_2026-06.md` §8 の確定済み設計判断「Google Sheets 永続化」を覆す提案**である。
   §1で整理した通り、採用理由(ステートレス回避)とCEOの今回の動機(セキュリティ負荷軽減)は
   別の問題意識だが、正典の明示的な上書きにあたるため**必須でCEO確認を要する**。
2. **「みんなの庭」タブのMVP版での非表示化**。原則には抵触しないが、既存の実装済み機能を
   公開版で見せない、というユーザーに見える形の判断であり確認を要する。
   (非表示であって削除ではない旨、CEOの意図と一致するか要確認)
3. **ログイン方式の変更(テスター選択制の廃止 → デバイスローカルの匿名識別子)**。
   公開に関わる仕様変更であり確認を要する。
4. **データ喪失リスクへの手当て(§3-4)の要否・優先度**。企画としては最低限の告知は推奨するが、
   エクスポート/インポート機能の実装可否・時期はCEO判断(開発工数とのトレードオフのため)。
5. 上記4点が承認された場合の**実装手法そのもの(ブラウザlocalStorageの具体的な実装方式・
   Streamlit上での実現可否)は開発部の技術判断領域**であり、本書はその可否を保証しない。
   企画からは「既存の `Storage` インターフェース設計(`ARCHITECTURE_PLAN_2026-06.md` §5)の
   延長線上にあり、方向として不自然ではない」という所見のみ述べる。

## 6. 開発部視点の規模感(参考・企画からの見積りではない)

- ストレージ抽象化: `ARCHITECTURE_PLAN_2026-06.md` §5 で既に設計済みの `Storage` Protocol に
  `LocalStorage`(ブラウザ側保存)実装を1つ足す形が最小の手戻りで済む方向性に見える
  (最終判断は開発部)。
- ログイン画面の撤去+匿名ID自動生成: 小規模(`render_login_screen` の削除・置換が中心)。
- 「みんなの庭」タブの非表示化: 極小規模(`app.py` のタブ定義から外すだけ。`community.py` は
  削除せず残置)。
- 音声配信(現状 xeno-canto の録音を base64 埋め込み)は、ローカル保存方針(サーバーに個人データを
  持たない設計)と相性が良い方向。ただしライセンス監査(`ROADMAP_GOOGLE_PLAY.md` の商用可否判断)
  は本提案とは別軸の既存論点であり、混同しないよう明記しておく。

## 7. まとめ

「基本ローカルデバイス保存」への転換は、**コア体験(3本の柱・受動のコアループ)には触れず、
永続化の場所とログイン方式だけを変える**提案として整理できる。唯一構造的に矛盾するのは
「みんなの庭」であり、これは★☆☆・可逆な実験という位置づけに従い**MVPでは非表示**とすることで
矛盾を解消する。交渉不能の5原則には抵触しないと判断するが、正典の確定済み判断を上書きする点、
および実装済み機能を非表示にする点は、`05_運用ルール.md` §1 に基づき**CEO確認を必須**とする。
