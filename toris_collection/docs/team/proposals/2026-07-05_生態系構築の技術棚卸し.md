## 2. 「植える→時間が経つ→鳥が来る」の時間換算の実態

### 2-1. 植物に「生育コスト(時間)」は現状ゼロ

`app.py` で植物を選ぶと、その場で `st.session_state.planted` に追加される
(`app.py:1711` 付近 `st.session_state.planted.append(pid)`)。`planted_at_map` に
植えた時刻(ISO文字列)を記録してはいるが(`app.py:1715`)、**この値を読んで
「まだ育っていないから食物網に参加しない」という判定を行う箇所はコード全体に存在しない**
(`_mature_for_evo` という変数名が `absence_loop._evolve_since_last_visit` にあるが、
中身は単に `list(st.session_state.get("planted", []))` であり、成熟判定はしていない
= 変数名が実態を先取りしているだけで、成熟ロジックは未実装)。

`engine.build_network()` は、植えた植物を気温適合度(`temperature_fit`)だけで
即座にネットワークへ組み込む(`engine.py:204-214`)。つまり **「植えた瞬間から、その植物は
虫・鳥にとって"存在する"ものとして扱われる**。現実の「苗が育つまで数年待つ」に相当する
遅延は、データ構造上もロジック上も一切ない。

### 2-2. 実際に時間換算されているのは「不在中に鳥が来る機会」だけ

時間が絡むのは `absence_loop.estimate_tick_count()`(`absence_loop.py:32-56`)のみで、これは
「植物の生育」ではなく「離れている間に鳥の到来・退去判定を何回まわすか」を実時間から
決めている:

```
< 5分      : 0サイクル
5-30分     : 1サイクル(コーヒー休憩)
30分-2時間 : 2サイクル
2-6時間    : 3サイクル
6-12時間   : 4サイクル
12-24時間  : 5サイクル
24時間以上 : 6サイクル(上限)
```

つまり現状の「待つ」体験は、**植物が育つのを待つのではなく、「(即座に完成した)食物網に
鳥が偶然立ち寄るのを、時間経過に応じた回数だけ抽選する」**という設計になっている。
月(`month`)は現実のカレンダー月(`datetime.now().month`、`app.py:505,581,761`)にひもづき、
季節による気温変化(`SEASON_TEMP_OFFSET`)は反映されるが、これも「生育の進み」ではなく
「今日は何月か」という到来確率の係数にすぎない。

### 2-3. 撹乱後の「再生」も現状は存在しない(重要な発見・ドキュメント不整合あり)

`HANDOFF_2026-06.md` §4-2 は `disturbance.py` に `roll_succession()`(倒れた跡地に別の植物が
芽吹く遷移ロジック)が実装済みと明記しており、直近の企画部提案書2件
(`2026-07-03_今すぐ追加するなら効果的な機能案3選.md` 案B、`2026-07-05_優先順位の妥当性検証.md` §1)
も、これを前提に「既存の `roll_succession`/`evolve_state` を再利用すればシミュ・ラボの
早送りは新しい確率式を作らずに実現できる」と評価している。

しかし **実際の `disturbance.py`・`tests/test_disturbance.py` には `roll_succession` が
存在しない**。`git log`/`git show` で確認したところ、2026-06-12 のコミット `851217d`
(「撹乱は植物を純減させる」)で **意図的に廃止済み**だった:

> 撹乱(嵐・落雷・伐採)で倒れた植物のあとに自動で別の植物を植え直す遷移
> (roll_succession / is_pioneer)を廃止。倒れた分は純減とする。自動植え直しは
> plantings を容量超過(例: 京都で 5/4 種)させ、同じ plant_id が重複して
> active になり得た。

つまり **現行の撹乱モデルは「倒れる(純減)」のみで「跡地に別種が芽吹く(遷移)」は
ない**。`test_disturbance.py` の `test_disturbance_story_branches` は
`"芽吹" not in s1` を明示的にアサートしており、これは仕様として確定している
(罰しない設計は保たれているが、"移ろい"の演出としての再生は現状無い)。

**`HANDOFF_2026-06.md` §4-2 はこの廃止を反映しておらず、古い実装内容のまま残っている**
(ドキュメント不整合)。この記述を根拠にした前回提案(「roll_succession を再利用すればよい」)
は、現状のコードでは前提が崩れているため、そのままでは成立しない。詳細は §4 で述べる。

## 3. 「土地の広さ」の抽象化(`max_plants`)

`data.py` の `BIOMES` を確認したところ、京都・シドニー・シャーロットの3バイオームは
**すべて `max_plants: 4`**(`data.py:19,28,37`)で、バイオームごとの差はない。
`app.py:1678` (`max_plants = biome_meta.get("max_plants", 8)`)で読み出され、
「植える」タブで **植えられる"異なる植物種"の数を4種に上限**する形で使われている
(面積・個体数ではなく「種数の枠」という抽象化。同じ種を複数植える概念はない)。

つまり現状の「土地の広さの制約」は、実際の面積・密度モデルではなく、**「同時に持てる
植物種は最大4」という単純な枠**であり、CEOの言う「土地の広さの制約」を、かなり
粗く(しかし明快に)抽象化したものと言える。3バイオームで差がない点は、
「土地の個性」を表現する余地がまだ使われていない(技術的には
`BIOMES[biome_id]["max_plants"]` を都市ごとに変えるだけで対応可能な、実装コストの低い拡張点)
ことも合わせて確認した。

## 4. 種の多様性 — データ構造上の拡張余地

`py -3` で実際にロードして確認したところ、現在のシード規模は
**植物43種・昆虫23種・鳥47種**(`00_共通サマリ.md`等に残る「35種」表記は既に古い)。

### 4-1. コード側に上限は無い
`engine.build_network()`・`run_turn()` は `PLANTS`/`INSECTS`/`BIRDS` という
素朴な `dict` を単純にループするだけで(`engine.py:218,230-238,322`)、件数に対する
ハードコードされた上限は存在しない。数百種に増えても計算量は線形にしか増えず
(1ターンあたりの計算はブラウザ内で一瞬で終わる規模)、**アーキテクチャ上のボトルネックはない**。
拡張のボトルネックは技術ではなく、**GloBIによる裏取り・xeno-cantoの音源・ドット絵アセットの
制作コスト**という「コンテンツ制作側」の制約に集約される。

### 4-2. ただし `species_loader.py` の「全置換」方式には注意が必要
`species_loader._load_all()`(`species_loader.py:184-205`)は、Google Sheets の
`species_birds`/`species_plants`/`species_insects` シートに1行でもデータがあれば、
**シード47/43/23種をまるごと置き換える**(シードとのマージではない)。つまり
Sheets経由で種を追加する運用を取る場合、既存の全種をシートに書き写した上で新種を
足す必要があり、「1種だけ追加」を意図した編集ミスで既存種が消える(=シートが空でない限り
シードにフォールバックしない)リスクがある。これは既存の設計(`species_loader.py`冒頭コメント)
どおりの仕様であり、バグではないが、拡張作業時の運用上の注意点として明記しておく。

## 5. シミュ・ラボ構想の再検証

### 5-1. 実装状況: 未着手。ただし類似目的の狭い機能が既に存在する
`CONCEPT_v2.md` の「シミュ・ラボ」(制約なし・早送りで `evolve_state` を回す別タブ)は
未着手のまま(`03_開発部.md`「未着手/配線待ちの差込点」の記載と一致)。

一方、`app.py` には既に **`🧪 シミュ`** という名前のタブがある(`app.py:1412-1414`,
`1771〜`)。中身を確認したところ、これは
`engine.calculate_arrival_probability()`/`build_network()` を使った
**「今の植生に、候補の植物を1つ足したら、特定の1種の到来確率がどう変わるか」を
その場で計算するだけの静的な比較ツール**であり、`absence_loop.evolve_state()` は
一切呼ばれない(=早送り・遷移・複数ティックの概念がない)。session_state の
`planted`/`residents` も書き換えない(比較用の一時計算のみ)ため、副作用がなく
安全という点はよい設計だが、CONCEPT_v2.md が言う「ラボ」とは別物である。
**同じ「シミュ」という名前が2つの異なる機能を指しうる**ため、実装時は名称の
区別(例:「シミュ(比較)」と「ラボ(遷移体験)」)を企画側で決めておく必要がある。

### 5-2. 前回見積り(2026-07-03案B)の妥当性再検証

前回提案は「中規模。新規タブUI + 早送り制御 + 遷移の可視化(`disturbance.py`の既存関数を
再利用)+ セッションのみで完結」としていた。今回コードを直接確認した結果、
**規模感の大枠(中規模)は妥当だが、「遷移の可視化」の前提が崩れている**:

- **早送り制御そのものは技術的に軽い**: `absence_loop.evolve_state()` は既に
  「ティック数だけ回す」設計になっている(`absence_loop.py:128` の `for i in range(n_ticks)`)。
  ラボ側は `estimate_tick_count()` の実時間換算をバイパスし、ユーザーが選んだティック数を
  直接渡すだけでよい。**新しい確率式は要らない**という前回の制約は、到来確率(`run_turn`)・
  撹乱(純減のみの`roll_disturbance`/`apply_disturbance`)については**今も成立する**。
- **しかし「遷移(succession)の可視化」は今のコードには実体がない**(§2-3)。
  「草地→低木→二次林と移ろう」という体験を見せたい場合、`roll_succession` を
  "再利用"することはできず、**新規に設計・実装する必要がある**。これは前回提案の
  重要な前提の誤り(HANDOFF記載の古さに起因)であり、今回の調査で訂正が必要。
- **ただし、ラボで再導入するなら元のバグを再発させない道筋がある**: `roll_succession`が
  2026-06-12に廃止された理由は「自動植え直しが `max_plants` を超過させた」という
  **本編(`plantings`シート・セーブコードに永続化される状態)特有の問題**だった。
  CONCEPT_v2.md のラボはそもそも「制約なし」(`max_plants` を適用しない)かつ
  「結果は本編に一切書き込まない」設計を想定しているため、**ラボ内限定で
  succession 相当のロジックを再実装しても、本編の容量超過バグを再発させる経路がない**
  (ラボの `planted` 配列は本編の `st.session_state.planted` と別変数にすれば済む)。
  つまり「遷移を見せる」機能自体は実現可能だが、**「既存ロジックの再利用」ではなく
  「ラボ専用の小さな新規ロジック」として設計し直す必要がある**、というのが本調査の結論。
- **月/季節も自由に動かせる**: `current_temperature(biome_id, month)` は既に
  `month` を引数に取る純関数(`engine.py:39-51`)なので、ラボでは実時間のカレンダー月
  (`datetime.now().month`)に縛られず、スライダー等で任意の月を選んで
  「真夏に植えたら」「真冬に植えたら」を試せる。これは追加の新規ロジック不要で、
  既存関数にラボ側から任意の `month` を渡すだけで実現できる。

### 5-3. ローカル保存(セーブコード)移行後の影響
`save_code.py` の `SAVE_KEYS`(`save_code.py:30-45`)は明示的なホワイトリスト方式であり、
ラボ用の新規 `session_state` キー(例: `lab_planted`, `lab_ticks_run`)を
**このリストに追加しなければ、何もしなくても自動的にセーブコードへ書き込まれない**。
「ラボの結果は本編のセーブコードに書かない」という設計方針は、
2026-07-04のローカル保存移行によって**むしろ実現しやすくなった**(以前のGoogle Sheets時代は
書き戻し関数を個別に分けて避ける必要があったが、今はホワイトリストに入れないだけで済む)。

### 5-4. 規模感の再結論
- 早送り制御・到来/撹乱(純減)の再利用・月操作: **小〜中規模**(既存関数の呼び出し方を
  変えるだけ、新規確率式なし)。
- 「遷移(succession)」の可視化を含める場合: **ラボ専用の新規ロジック設計が追加で必要**
  (前回見積りの「中規模」に、遷移ロジックの新規設計・パイオニア種指定データの整備分が
  上乗せされる。ただし本編の容量超過バグとは無関係な形で作れるため、リスクは低い)。
- 「遷移を見せない(到来・撹乱の早送りのみ)」スコープに絞るなら、前回見積りの
  中規模のまま、追加ロジックなしで着手できる。

## 6. 技術的に何ができて何が難しいか(まとめ)

**すぐできる(既存の延長・新規ロジック不要)**
- ラボでの早送り(`evolve_state` のティック数を直接指定、新しい確率式は不要)。
- ラボでの月・季節の自由操作(`current_temperature` は既に純関数)。
- バイオームごとの `max_plants` の差別化(現状3都市とも4で横並び。値を変えるだけ)。
- 種数の拡張(データ構造上の上限なし。ボトルネックはGloBI裏取り・音源・ドット絵の制作)。
- ラボの状態を本編セーブコードに書かない設計(`SAVE_KEYS`に加えないだけで達成)。

**設計をやり直す必要がある(既存の"再利用"では済まない)**
- 「遷移(succession)」の可視化。`roll_succession` は本編から意図的に廃止済みであり、
  復活させるなら新規設計が要る(ただしラボ限定にすれば元のバグ要因とは無関係)。
- 「生育を待つコスト」を本編にも部分的に持ち込みたい場合(例: 植えてからN時間/N日は
  食物網に参加しない、等)は、`build_network`/`_evolve_since_last_visit` に
  成熟判定を新設する必要があり、**受動性(原則1)と両立するバランス設計
  (急かしにならない遅延のかけ方)を企画部と詰める必要がある**(交渉不能の原則1に
  抵触しない設計になっているか、実装前にCEO/企画部の確認を要する領域)。

**ドキュメント側の要対応**
- `HANDOFF_2026-06.md` §4-2 の `roll_succession` 関連記述は現状のコードと乖離しており、
  今後の混乱を避けるため部長経由での是正(または注記追加)を推奨する。

- 「生育を待つコスト」を**本編**に新たに持ち込む変更は、コアループ(離れる→時間が進む→
  発見)の体感に直接影響するため、`05_運用ルール.md` §1 の「コアループの大幅な作り直し」

- シミュ・ラボの新規タブ追加は「既存タブ構成」への追加にあたるため、着手時に
  `05_運用ルール.md` §1 に基づき計画提示→承認が必要(本書はその前段の技術材料提供)。
- `HANDOFF_2026-06.md` の是正は正典ドキュメントの更新であり、内容に実質的な設計変更を
  伴わない事実修正のため、通常は部長判断で反映可能と考えるが、念のため報告に含める。

## まとめ(300字程度)

現状、「植える→時間が経つ→鳥が来る」の"待つ"は植物の生育ではなく、離れた実時間を
到来抽選の回数に変換する仕組み(absence_loop）のみで表現されており、植物は植えた瞬間に
食物網へ即参加する(生育コストは実質ゼロ)。土地の広さは3都市とも「植物種4つまで」という
同一の枠で抽象化。種数(植物43/昆虫23/鳥47)はデータ構造上の上限がなく拡張は容易(制約は
アセット制作側)。シミュ・ラボは未着手で確認どおりだが、前回提案が前提にした
「roll_succession の再利用」は2026-06-12に本編から意図的に廃止済みで実在しないため、
遷移(succession)演出は新規設計が必要(ラボ限定なら元の容量超過バグとは無関係に作れる)。
早送り自体・月操作は既存関数の再利用で新しい確率式なしに実現可能。
