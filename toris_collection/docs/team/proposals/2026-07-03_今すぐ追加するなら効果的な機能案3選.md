## 案A: おやつ(餌台)の本編UI配線 — 「置く=関係を1本足す」

### 1. 概要
`feeder_chain.py`(FEEDERS/ANIMALS/RAPTORS + `resolve()`/`wary_arrival_multiplier`)は
実装・テスト済みだが UI が未配線。以下を追加する。

- 「植える」タブ(または新設の小さな「おやつ」セクション)に、**消費されないノード**として
  おやつ(木の実ケーキ・花蜜・スエット等)を置ける UI を追加。
- 置いたおやつは `run_turn`/不在ループの前に `feeder_chain.resolve()` を通し、
  `wary_arrival_multiplier` を到来確率に乗算(= 意外な・レアな来訪の可能性が少し上がる)。
- おやつの入手は**無料資源**(プレイで貯まる、例: 会った日数や落とし物の副産物)。
  課金は行わない設計を基本とし、有料化する場合は**見た目(スキン)のみ**に限定する
  (下記「原則チェック」参照)。
- リス・猛禽などは `role` により図鑑・ラジオから除外(既存方針どおり、収集対象に混ぜない)。
- 数値(到来確率・乗数)は**表示しない**(HANDOFF §8「進捗バー・警戒度数値の非表示」を踏襲)。
  ユーザーには「このおやつは種子を食べる鳥を呼びやすくします」など定性的な説明のみ見せる。

### 2. なぜ「今すぐ」効果的か
- **実装コストが最小**: ロジック・テスト(`tests/test_feeder_chain.py`)が既にあり、
  配線点も `03_開発部.md` に明記済み。UIを足すだけで新しい遊びの層が生まれる、
  投資対効果が最も高い項目。
- **未回収の投資を回収する**: 既に書かれたコードが使われていない状態は、企画・開発の間の
  「積み残し」。ここを塞ぐことは次の機能追加より優先度が高いと判断。
- CONCEPT_v2.md の「小さく出す順」③、`01_企画部.md`「次に詰めること」②に該当し、
  両ドキュメントが揃って推している項目。
- 現状の課題(`04_社長.md`)である「継続率・滞在時間の観測」に対し、
  「今日は何を置こうか」という**軽い能動的選択**を追加することで、開いたときの
  やることの解像度が上がる(受動性は壊さない範囲で)。

### 3. 交渉不能の5原則チェック
1. **受動的か**: 条件付きでYes。おやつは**消費されない・毎日の補充を要求しない**ノードとして
   設計する必要がある(CONCEPT_v2.md に明記の設計)。もし「時間が経つと効果が切れて置き直しが
   必要」等の消費型にすると、水やり的な"急かし"になり原則1に抵触する。**必ず非消費・恒常設置**
   として実装すること。
2. **罰しないか**: Yes。おやつを置かなくても既存のコレクションは減らない。撤去しても図鑑・会った
   日数への影響はない。
3. **鳥の声と癒しは常に無料か**: 条件付きでYes。基本の呼び込み効果(`wary_arrival_multiplier`)は
   **無料資源のみで完結**させる。有料版を作る場合は「見た目の違うおやつスキン」に限り、
   効果(呼び込み力)は無料版と同一にする。効果に差をつけると「課金しないとレアな鳥の声に
   出会いにくい」という事実上のpay-to-winになり、原則3に抵触するため**明確に禁止**する。
4. **生態に誠実か**: Yes。`feeder_chain.py` は既存の役割定義(FEEDERS/ANIMALS/RAPTORS)に基づき、
   恣意的な「なぜ来たか」を追加しない。
5. **かわいさ最優先か**: Yes、ただし要注意。おやつ自体のドット絵・設置アニメーション・
   リス/猛禽が来た際の一瞬の演出まで「かわいさ」の水準を保つ必要がある(手を抜くと世界観の
   質が下がる)。既存の `SPRITE_ALIASES` 流用で最初は賄える。

**結論**: 5原則すべてクリア可能。ただし「非消費」「効果は無料資源のみ」の2点は**設計上の
必須制約**として明記し、崩さないこと。

### 4. 実装規模感(開発部視点、参考)
小〜中規模。既存ロジック・テストが揃っているため、主な作業は
(1) UI(設置・撤去の操作、置いてあるおやつの一覧)
(2) `run_turn`/`absence_loop` への `feeder_chain.resolve()` 差し込み
(3) おやつ・来訪動物のドット絵(既存アセット流用でMVP可)
(4) role によるラジオ/図鑑からの除外の確認
(5) 既存テストの拡張。
「壊さない検証」観点でも、既存コアループに追加で乗せる形なので影響範囲は限定的。

## 案B: シミュ・ラボ(制約なし・遷移プレビュータブ)

### 1. 概要
本編とは別の新規タブ(または画面)として、`engine.py`/`absence_loop.py` を
**土地・時間・資源の制約なし**で早送り実行できる「実験場」を追加する。

- 「何を組むとどう遷移するか(草地→低木→二次林)」「その結果どんな鳥が来るか」を、
  待たずに試せる。
- ラボの結果は**セッション内のみで完結**。本編の `会った日数`・図鑑・Google Sheets には
  **一切書き込まない**(CONCEPT_v2.md・03_開発部.md で明記済みの設計方針)。
- 表示は既存のドット絵・鳥のアイコンを流用し、データ的な無機質さを避ける。

### 2. なぜ「今すぐ」効果的か
- `01_企画部.md`「次に企画で詰めること」の**最優先項目**であり、CONCEPT_v2.md の
  「小さく出す順」でも②(日次カウントの直後)に位置づけられている、いわば次の既定路線。
- 3本の柱のうち「なぜ来たかが関係でわかる(学び)」を、**受け身の発見待ちでなく
  能動的に探究できる形**で補強する。現状は「たまたま来た鳥から逆算的に学ぶ」しかできず、
  好奇心が強いユーザーの学習欲を受け止めきれていない。
- 差別化(Merlin=識別ツール、Birdfull=収集idle)に対し「生態の理解が攻略になる」という
  Torisの堀を、最も明示的に見せられる機能。

### 3. 交渉不能の5原則チェック
1. **受動的か**: ここは唯一「任意・能動的な実験レイヤー」であることを明記する必要がある
   (HANDOFF §4-4 のレイヤー表で「○ 能動(任意)」と整理済み)。本編の受動性を代替・強制しない
   限りにおいて問題ない。**ラボにログインボーナスや連続日数条件を絶対に付けないこと**
   (付けると原則1に抵触)。
2. **罰しないか**: Yes。ラボはセッションのみで結果が保存されないため、そもそも「失う」ものが
   ない。
3. **鳥の声と癒しは常に無料か**: Yes、ただし要設計。ラボ内で試しに聴ける鳴き声も無料とし、
   将来的にも課金要素をラボに持ち込まないことを明記する。
4. **生態に誠実か**: 最重要チェックポイント。早送り演出のために遷移ロジックを簡略化しすぎると
   `disturbance.py` の既存モデル(種数-面積関係 S=cA^z 等、根拠のある式)と矛盾する恣意的な
   近道を作ってしまうリスクがある。**既存の `roll_succession`/`apply_disturbance` をそのまま
   再利用し、"早送り"はティック回数を増やすだけ(新しい確率式を作らない)**という制約を設ける。
5. **かわいさ最優先か**: 要注意。「ラボ」という名称・実験UIは無機質になりやすい。
   グラフ/表だけの画面にせず、既存のドット絵・簡単なアニメーションで「遷移していく庭」を
   見せる演出は最低限維持すること。

**結論**: 5原則クリア可能だが、「早送り時に新しい確率式を作らない(既存モデルの再利用に
限定する)」という制約と、「かわいさを保つ表現に予算を割く」という2点が設計上の留意点。

### 4. 実装規模感(開発部視点、参考)
中規模。新規タブUI + 早送り制御(ティック数を指定して `evolve_state` を回す)+
遷移の可視化(`disturbance.py` の既存関数を再利用)+ セッションのみで完結させ Sheets に
書き込まないことの確認(テストで担保)。既存の「みんなの庭」(`community.py`)を
新規タブとして追加した際の規模感に近い。案Aより工数は大きいが、新規インフラは不要。

## 案C: 「会った日数」の節目演出 + デジタル特典への橋渡し

### 1. 概要
「会った日数(1日1回)」は直近で実装済みだが、`01_企画部.md` は
「節目(10/30/100)と称号、通知・図鑑での見せ方」を**未解決の論点**として明記している。
つまりデータ基盤はできているが、報酬としての見せ方がまだ薄い。以下を追加する。

- 節目(10日/30日/100日)を跨いだときに、図鑑上で**静かな演出**(バッジ・一瞬のアニメーション・
  「常連になりました」的な一言)を表示。プッシュ通知ではなく、**次に開いたときに気づく**形
  (HANDOFF §8「通知機能なし」を踏襲、プッシュは使わない)。
- 節目ごとに、その鳥の**豆知識カード**(自前デジタル特典、CONCEPT_v2.md「収集→交換」①)を
  1枚解錠。内容は GloBI 由来の食性・関係性など事実ベースの情報に限定(恣意的な逸話は作らない)。
- (任意・後回し可)節目時に「◯◯と会って30日」の共有カード画像を生成し、SNS部の
  既存フォーマット案(`02_SNS部.md`「会った日数」欄)にそのまま接続できるようにする。

### 2. なぜ「今すぐ」効果的か
- **最も新しく入った機能(会った日数)の"仕上げ"**であり、既にあるデータ(`load_bird_days`)を
  そのまま使える。新規の重い実装がほぼ不要。
- `CONCEPT_v2.md`の「収集→交換」ロードマップの**最初の一歩**を、新規インフラ(DB移行等)
  なしで先取りできる。
- 企画部(節目の見せ方)・SNS部(共有素材)・社長(継続率のKPI可視化)の3部署が同時に
  得をする、依存関係の少ない施策。
- `04_社長.md`の「習慣化装置(不在ループ+会った日数)」という勝ち筋を、いちばん薄い部分
  (節目の可視化)から厚くする。

### 3. 交渉不能の5原則チェック
1. **受動的か**: Yes。節目判定は既存の**累計・非連続**の会った日数カウント
   (`sheets_client.load_bird_days`、1日1回・日をまたげば途切れても損しない設計)に
   完全に乗る。**「連続ログイン◯日」のような streak 型に変えないこと**(変えると
   「途切れると損」になり原則1に抵触)。現行設計のまま拡張すれば問題ない。
2. **罰しないか**: Yes。会った日数は既存仕様上、撹乱で庭が痩せても減らない(HANDOFF §1-1-a
   のラジオ=蓄積するコレクションと同じ思想)。本提案はこれに手を加えない。
3. **鳥の声と癒しは常に無料か**: Yes、要設計。解錠する豆知識カード・共有カードは無料。
   将来「サポーター向け高解像度エクスポート」等を検討する場合も、**基本の解錠・閲覧は無料**
   のまま、付加価値(彩り)のみ課金対象にする。
4. **生態に誠実か**: 要注意点あり。豆知識カードの文面は**GloBI由来の事実**または出典のある
   情報に限定し、鳴き声のクレジット表記と同様に**出典を明記**する。作り話の「なぜ来たか」逸話
   は原則4に抵触するため厳禁。
5. **かわいさ最優先か**: Yes。節目演出は数値バッジだけでなく、鳥のドット絵・鳴き声を主役にした
   小さな祝祭演出にすること(単なるカウンター表示にしない)。

**結論**: 5原則クリア可能。ただし「非連続カウントの現行仕様を維持」「豆知識は事実ベース+出典
明記」の2点は必須制約として明記。

### 4. 実装規模感(開発部視点、参考)
小〜中規模。節目判定ロジックは既存カウントの薄い拡張(小)。豆知識カードは**データ入力作業**
(コード的には軽いが、種ごとの文章準備が要る。まず主要種数種から始めて拡張可)。
共有カード画像の自動生成は `02_SNS部.md` が「企画中(自動生成が理想)」と位置づける項目で、
画像合成の実装が絡むためやや重め。**MVP(図鑑内の節目バッジ+豆知識解錠のみ)に絞れば小規模**、
共有カード生成まで含めると中規模。

## 推奨優先順位と理由

**A(おやつ配線)→ C(節目演出)→ B(シミュ・ラボ)** の順を推奨する。

- Aは実装コストが最小で、既存の未回収投資(ロジック実装済・UI未配線)を最速で回収できる。
- Cも既存データを使う低コスト施策で、3部署に同時に効き、収集→交換ロードマップの布石になる。
- Bは効果(生態理解の深化・差別化訴求)は大きいが、UIとしての「かわいさ」の担保に企画・開発
  双方の追加検討が要るため、A・Cよりやや重い。ただし`01_企画部.md`の最優先項目でもあるため、
  A・Cと並行して仕様の詰め(遷移の見せ方等)を進める価値はある。
