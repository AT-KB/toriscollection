# MVPローンチ → マネタイズ戦略・ロードマップ・1ヶ月計画

作成: 2026-06 / 方針は CONCEPT_v2.md、社訓は docs/team/00_共通サマリ.md に従う。
合言葉: **社訓を守りつつ、とにかくまず MVP でローンチ**。

## 0. 社訓(これだけは絶対に守る)
- 受動的(急かさない・罰しない) / 鳥の声と癒しは常に無料 / 生態に誠実 / かわいさ最優先。
- MVP でも上を1つも破らない。破るくらいなら機能を削る。

---

## 1. 戦略(マネタイズまでの3段)

**段階収益化。作り込みでなく"出して観測して改善"。**

| 段 | 目的 | 収益 | 判断ゲート |
|---|---|---|---|
| **S1 ソフトローンチ(今月)** | 動くものを公開し、継続率と滞在時間を測る | 無し(計測に集中) | 7日継続 / 1セッション滞在 が取れるか |
| **S2 マネタイズ点火(1〜2ヶ月後)** | 広告+サポーター課金を"静かに"入れる | 広告・小額IAP | ARPDAU と 離脱率(広告で減らないか) |
| **S3 実世界の柱(3ヶ月+)** | コラボ/優待・教育で広告以外の収益 | アフィリエイト/コラボ | 継続ユーザーの母数 |

**マネタイズの前提(先に潰す)**: 広告=商用 → 鳴き声は商用可(CC0/BY)に絞る。
→ S1 中に **ライセンス監査**を回して「商用で何種鳴くか」を確定する(これが S2 の入口条件)。

---

## 2. MVP の定義(スコープを固定する)

### MVP に「入れる」= もうほぼ在る
今の実装で MVP の中身は概ね揃っている。**新規開発を増やさず、公開品質に仕上げる**のが今月の主眼。
- バイオーム構築(植える)/ ラジオ(実声)/ ヒーリングBGM / 図鑑・落とし物 /
  不在ループ / 会った日数 / みんなの庭 / 京都・シャーロット。

### MVP から「外す」(後回し・迷ったら削る)
- シミュ・ラボ、おやつ課金、収集→交換、餌台UI配線、通知(PWA)、Google Play、DB移行、
  新バイオーム、専用ドット絵の刷新。→ **全部 S2 以降**。

### MVP の合格条件(公開してよい状態)
1. 未ログイン/初回でも 3タップ以内に「鳥が鳴く」まで到達できる。
2. 主要タブでエラーが出ない(AppTest 例外0・実機スモーク済)。
3. プライバシーポリシーと素材クレジット(xeno-canto等)が表示される。
4. 「使い方」が1画面で分かる(受動性=放っておくと来る、が伝わる)。

---

## 3. ロードマップ(3ヶ月ダイジェスト)

```
今月(S1): 公開品質仕上げ → ソフトローンチ(Web) → 計測開始
  ├ ライセンス監査で商用可カバー率を確定
  ├ オンボーディング/使い方/クレジット/プライバシー
  └ 計測(継続率・滞在・会った日数の伸び)を仕込む

翌月(S2): マネタイズ点火
  ├ 静かなバナー1枚 + 任意リワード広告(声はタダのまま)
  ├ サポーター課金(広告オフ+バッジ)
  └ commercial_only を ON(監査結果次第)

3ヶ月+(S3): 拡張と実世界
  ├ シミュ・ラボ / おやつ=餌台ノード / 収集→交換
  ├ PWA 通知 → TWA で Google Play(12人×14日テスト)
  └ DB 移行(伸びたら) + コラボ/優待
```

## 4. 今月(4週間)計画 — MVPローンチ

> 前提: 開発は主に「仕上げ・計測・公開準備」。新機能は増やさない。各週末に"出せる状態"を保つ。

### Week 1 — 前提潰し & 計測設計
- [ ] **ライセンス監査**を Cloud で実行し、商用可カバー率と「NCのみ/録音なし」種を確定。
- [ ] 計測方針を決める(継続率・1セッション滞在・会った日数・タブ別離脱)。
      まずは既存 access_logs / bird_visits から算出できる範囲で開始。
- [ ] **KPI 目標を仮置き**(下記 §5)。
- [ ] プライバシーポリシー草稿 + 素材クレジット文面(xeno-canto/Freesound/ライブラリ)。

### Week 2 — オンボーディング & 公開品質
- [ ] 初回導線: 未ログイン/初訪問でも「植える→数十秒で鳴く」体験に最短で乗せる。
- [ ] 「使い方」を受動性が伝わる1画面に(放っておくと鳥が来る/声は無料/庭は不要)。
- [ ] 空状態の文言(鳥ゼロ/植物ゼロ時)を親切に。エラー時のフォールバック確認。
- [ ] 実機スモーク(スマホ幅)で主要タブとラジオ再生を通す。

### Week 3 — 仕上げ & 事前告知
- [ ] バグ潰し(AppTest 例外0・全テスト pass を維持)。ラジオの間/音量の最終調整。
- [ ] ストア/共有用のビジュアル(アイコン・スクショ・一言コピー)を SNS部と用意。
- [ ] 少人数のクローズド配布(友人・テスター)で初期フィードバック。
- [ ] プライバシーポリシー公開(静的ページ)。

### Week 4 — ソフトローンチ & 観測
- [ ] **公開(Web)**。SNS部が「かわいい/癒し/軽い学び」で告知(誇張・課金誘導なし)。
- [ ] 計測を見る(継続率・滞在・会った日数)。定性フィードバックを集める。
- [ ] レトロ: 数値と声から「S2でマネタイズ点火してよいか」を判断。
- [ ] S2 バックログ(広告・課金・commercial_only)を確定。

---

## 5. KPI(仮の目安・S1で検証)
- **翌日継続 ≥ 25% / 7日継続 ≥ 10%**(癒し系の目安。低ければ導線か"来る頻度"を疑う)。
- **1セッション滞在 ≥ 90秒**(BGM文脈なら伸びる。広告インプレの土台)。
- **会った日数の伸び**: 継続ユーザーの平均"のべ会った日数"が週次で増える。
- S2 点火条件: 上記が横ばい以上 かつ 広告導入で翌日継続が大きく落ちないこと。

---

## 6. マネタイズ設計(S2で入れる・社訓遵守)
- **広告**: ホーム下部に静かなバナー1枚(ラジオ再生中は隠す)+ 任意リワード
  (「見ると今日は珍しい鳥が来やすい」)。全画面割り込みは無し。
- **課金**: サポーター(1回・小額=広告オフ+バッジ+図鑑フル)。ガチャ/スタミナ/時短なし。
- **鉄則**: 声と癒しは無料のまま。広告は"体験の外周"に置き、癒しの中心を汚さない。
- **前提**: `commercial_only=true`(監査後)。使えない種は当面ドット絵/気配で出す。

---

## 7. リスクと対処(MVP段階)
| リスク | 対処 |
|---|---|
| 鳴き声の権利で使える種が減る | 監査で事前把握 → 商用可に絞る。足りない種は絵で先行 |
| 誰も"来る頻度"を体感できない | オンボーディングで初回に早めの来訪を演出(受動性の証明) |
| Sheets の限界(伸びたら) | S1は小規模で問題なし。伸兆が出たら S3 で DB 移行 |
| 広告で癒しが壊れる | S2 は静かなバナー+任意リワードのみ。数値で離脱を監視、ダメなら退く |

## 8. 一言
**「まず出す。社訓は削ってでも守る。数字を見て、静かにマネタイズを足す。」**
今月のゴールは"完璧"でなく"観測できる公開"。
