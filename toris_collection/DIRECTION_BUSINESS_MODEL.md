## 一行で

**ねこあつめ／ピクミンBloom の"受動的な癒し"に、「日常が反映されて鳥が来る」と
「生態系が自然に学べる」を掛けた、広告で副収入を得るアプリ。がっつり課金はしない。**

- 受動的: 放っておくと時間が進み、鳥が来る。急かさない・罰しない。
- 日常連動: 実際の時刻・季節・(将来)天気・再訪リズムが庭に反映される。
- かわいい: 鳥が来る瞬間 / 鳴き声 / ドット絵。ここが心臓。
- 学べる: なぜその鳥が来たか(食物網)、渡りの時期が現実と一致。押しつけない。

Web アプリのままで始めてよい(TWA で後から Play にも出せる)。

## 0. 最重要の分岐:広告 = 商用。鳴き声ライセンスと衝突する

**これを決めないと先に進めない。** 広告を載せた瞬間、アプリは「商用」になる。
ところが今使っている xeno-canto の録音は **BY-NC(非商用)系が多数**で、
商用アプリでは使えない録音が大量に出てくる。

選択肢(推奨は A):

- **A. 商用可能な音源だけに絞る(推奨)**
  xeno-canto の中でも **CC BY / CC0** の録音だけを使う。Freesound の CC0 も併用。
  → コード側に「ライセンス列で商用可のものだけ採用」フィルタを足す(citation 取得は既にある)。
  → 一部の鳥は鳴き声が手に入らない → その鳥は「ドット絵+シンセの気配」で当面出す。
  → まず監査スクリプトで「今鳴いている録音の CC 内訳」を出し、影響範囲を可視化する。
- **B. 鳴き声を自前で用意**: 品質最高だが収録コスト大。将来の目玉として少しずつ。
- **C. 広告をやめて寄付/サポーター課金だけ**: NC 録音を使い続けられるが、副収入は細い。

> 結論: **A で始める**。「かわいい鳴き声」は資産の核なので、商用可の範囲で確保する。
> 監査の結果しだいで B を部分的に足す。

## 1. 収益モデル(受動的・低圧の範囲で)

「がっつり課金しない」= **広告は控えめ、課金は任意の小額だけ**。癒し系の世界観を壊さない設計。

| 手段 | 具体 | 世界観との相性 |
|---|---|---|
| **リワード広告(任意)** | 「見ると今日だけ珍しい種が来やすくなる/特別な種を1つ植えられる」 | ◎ ユーザーが選ぶので不快感が少ない。ねこあつめの煮干し的 |
| **静かなバナー** | ホーム下部に1つだけ。ラジオ再生中は隠す | ○ 置き場所を選べば許容。全画面割り込みは× |
| **サポーター課金(1回・数百円)** | 広告オフ + 「サポーターの庭」バッジ + 落とし物図鑑フル | ◎ 応援したい人向け。ガチャ課金はしない |
| **季節イベント(無料)** | 現実の季節に合わせた限定の鳥・装飾。課金なし、来訪動機 | ◎ リテンションの主役 |
| **教育コラボ(将来)** | 野鳥の会・博物館・自治体とタイアップ。広告以外の収入と信頼 | ◎ 「生態系が学べる」強みが効く |

やらないこと: 全画面割り込み広告、スタミナ課金、ガチャ、時短課金。**急かす設計は全部やらない。**

## 2. 「日常連動」を Web でどこまでやれるか(受動性の核)

ピクミンBloom は歩数/GPS を使う(ネイティブ)。Web は制約があるが、**現実を映す**という体験は十分作れる。
効果と実装しやすさで並べる:

1. **時刻(実装済み・今すぐ強化可)**
   朝はさえずり、夜は静か。朝チャイム的に「朝の庭」が賑わう。→ 今の実時刻対応を演出に格上げ。
2. **季節・日付(実装済み)**
   現実の季節と渡りを一致させる(ツバメは春夏だけ来る等、既にある)。「今週から冬鳥」。
3. **天気連動(新規・効果大)**
   ブラウザ位置情報 + 無料天気API。雨の日は水浴びの鳥、晴れは活発、など。
   → 「今日の自分の街の天気」が庭に出る = 最も"日常が反映された"感が出る。
4. **再訪リズム(実装済みの不在ループ)**
   留守が長いほど色々起きている。連続来訪はご褒美だが**途切れても罰しない**(今の思想を維持)。
5. **Web プッシュ通知(PWA・Android で可)**
   「🐦 あなたの庭にツバメが来ています」。受動アプリの再訪エンジン。
   → iOS Safari は制限あり。Android Chrome / TWA では有効。
6. **(将来・弱め)歩数/端末モーション**
   Web の歩数取得は不安定。核にはしない。将来ネイティブ化したら Pikmin 的に。

> Web でも 1〜5 で「日常が反映 → 鳥が来る → 鳴き声かわいい → なぜ来たか学べる」の輪は回せる。

   → 鳥が来る(留守中の来訪)
     → 戻ってきて発見 = かわいさの報酬(飛来アニメ/鳴き声/ドット絵)

- **みんなの庭**(既存の集合地図)は共感の装置。「隣の庭にも来てる」= 生きた世界。

## 2.5 方針更新: 天気連動は「見送り」。代わりの受動フック案

天気・水浴び・(現実)季節の作り込みは **負荷のわりにリターンが薄い**ため見送る。
代わりに、**今ある資産を演出に変えるだけ**で効く受動フックを優先する
(新規APIも位置情報も不要 = 低コスト高リターン)。

1. **留守中の「落とし物」を報酬の主役にする(最優先・ねこあつめの心臓)**
   既に mementos がある。「留守中にヤマガラが羽根を置いていった」を、
   戻ったときの**開封演出**にする。ねこあつめの"煮干し/グッズ"に相当。
   → 開くたびに小さなサプライズ = 受動再訪の最大の動機。実装資産あり。

2. **朝と夜で顔ぶれ・賑わいが変わる(実時刻は実装済み)**
   朝は合唱で賑やか、昼はまばら、夜は数種が静かに。
   → 「朝の庭」「夜の庭」で1日2回開く習慣。演出の格上げだけで作れる。

3. **珍客(レア来訪)**
   既存の確率エンジンで、たまに珍しい鳥。戻ったら「今日は珍しいお客さんが来た」。
   → ねこあつめのレア猫。集める動機と話題性(共有カード)に直結。

4. **鳴き声の"聞き比べ"コレクション**
   かわいい鳴き声が核。さえずり/地鳴き/個体差を図鑑で集める。
   「この鳥のこの鳴き方、まだ聴いてない」= 収集の縦の深み。

5. **名前をつける・会った回数で愛着(群れ演出は実装済み)**
   よく来る鳥にあだ名。回数で群れが増える(既存 flock)。愛着 = 継続。

6. **ヒーリングBGMモード(滞在時間=広告リターンに直結)**
   集めた鳥+環境音を**流しっぱなし**にできる作業用/就寝用モード。
   受動 × 癒し × 広告収益(表示時間が伸びる)と最も相性が良い。

> 共通する狙い: **「留守 → 戻る → 小さな驚き(落とし物・珍客・新しい鳴き声)」**の
> 循環を、新規負荷ゼロ〜小で濃くする。これが受動アプリのリテンションの本体。

## 8. コアの再設計:「庭づくり = 誘致と排除のデザイン」(餌箱の発想)

### 発想の出どころ
アメリカの裏庭バードウォッチング: 餌台を置き、セーフフラワーの種にして
リスや乱暴な鳥を避け、狙った小鳥を呼ぶ——**「何を差し出し、何を防ぐか」の
デザイン**そのものが趣味になっている。これを本アプリの食物網エンジンに載せると、
**遊びながら本物の生態知識が深まる**。

### いまのモデルと、足す一層
- 現状: 植物を植える → (GloBI由来の)食物網で 植物→昆虫→鳥 が創発 → 鳥が来る。
  バイオーム(京都/シドニー/シャーロット)が種プールと気温を決める。
- **足す一層 = 誘致と排除のトレードオフ**。庭に置くもの一つひとつに、
  「呼ぶ鳥」と「呼んでしまう厄介者」の両面を持たせる。プレイヤーは
  **狙った鳥が来る庭を"設計"する**ことになる。既存の `wariness`(警戒心)特性が鍵。

### 置けるもの(植物に加えて「庭の装置」を導入)
| 装置 | 呼ぶ | 副作用(厄介者・トレードオフ) | 学べること |
|---|---|---|---|
| 餌台(開放型) | スズメ・カワラヒワ等の種食鳥 | カラス/ムクドリ/**リス**が独占し、警戒心の強い小鳥が来なくなる | 開放給餌は優占種を呼ぶ |
| 餌台(かご型/セーフフラワー) | 小型の種食鳥だけ | 大型・リスを排除 | 給餌方法で来る種を選べる |
| 水場 | カワセミ・水浴びする鳥・両生類 | 蚊など不快な昆虫も増える | 水は多様性を上げるが管理が要る |
| 巣箱 | シジュウカラ等の樹洞営巣性 | 場所を巡る競争 | 営巣資源が繁殖=再訪を生む |
| 低木の茂み(既存: ヤブツバキ/ナンテン) | 警戒心の強い鳥(ウグイス等)に隠れ場所 | — | 隠れ場所が臆病な種を呼ぶ |
| 在来の実・木の実(既存 PLANTS) | 専門食の鳥。餌台より"誠実"な生態 | 実る季節に依存 | 在来植生の価値 |

### 仕組み(既存エンジンに素直に載せる・恣意的な数値を足さない)
1. **優占種/哺乳類の"圧"**: 開放型の餌が置かれると、優占種(カラス等)や **リス** が来やすくなる。
   これらが庭にいる間は、**`wariness` の高い鳥の到来確率が下がる**(既存の到来確率に係数を1つ)。
   → 「厄介者を排除すると臆病な小鳥が来る」= 餌箱ゲームの核を、警戒心特性で自然に表現。
2. **誘致の対象化**: 装置ごとに「呼ぶ種」を食物網の言葉で定義(GloBI で裏取り)。
3. **"なぜ来たか"の言葉を設計語に**: 到来理由を「あなたが低木で隠れ場所を作ったので、
   警戒心の強いウグイスが来ました」のように、**プレイヤーの設計判断に紐づけて**返す
   (理由生成は既にある)。これが「学び」の実感になる。

### なぜ良いか
- ねこあつめの"置いて待つ"に、**選択の意味(生態)**が乗る = 差別化かつ受動性は維持。
- 収集(鳥・鳴き声・落とし物)に **設計の達成感**が加わる。
- リス/優占種という"敵役"がいることで、共有・話題(「今日はリスに餌台占領された」)も生まれる。

### 段階導入(小さく出す)
1. まず **リス(または優占種)1体** + **餌台(開放/かご)** の2択だけ入れる。
   開放だとリスが来て小鳥が減る、かごだと来ない——最小のトレードオフ体験。
2. 反応が良ければ 水場・巣箱・茂みへ拡張。

## 9. 鳥の種類を増やす(GloBI ベース)

### 方針
- **バイオーム(京都/シドニー/シャーロット)に実在する種**から選ぶ(生態的に誠実)。
- 食性は **GloBI(globi_client.get_diet)で裏取り**し、既存の PLANTS/INSECTS と接続。
  足りない食物は PLANTS/INSECTS 側も少し足す。
- 1種あたり必要: name/scientific/english/color/eats_plants/eats_insects/temp_fit/
  biome_pref/rarity/wariness/description(+ 可能なら designbird のドット絵)。
- **ドット絵がボトルネック**: スプライトが無い鳥は色ドット表示になり「かわいい」が弱まる。
  → データ追加とスプライト追加は分けて進め、**スプライトが用意できた種から本採用**する。

### 進め方(半自動)
1. 候補の学名リストを作る(各バイオーム 5〜10種)。
2. ヘルパースクリプトで GloBI から食性を取得し、**data.py 追記案を生成**(要キュレーション)。
3. 人が確認して data に追加 → スプライトを用意 → 図鑑・ラジオに載る。
   ※ このリポジトリには GloBI クライアントとキャッシュが既にあるので、
     ネットワークが通る環境(デプロイ先/ローカル)でスクリプトを回す。

> まずは「候補学名リスト + GloBI 取得スクリプト」から。どのバイオームを優先するか教えてください。

## 11. 餌台・リス・水場・Hawk を「GloBI で」組むロジック

> 要望: 恣意的な"なぜ来たか"ロジックは作らない。**中心はあくまで GloBI のバイオーム相互作用**。
> 餌台はリスを呼び、リスは Hawk を呼ぶ——この連鎖も GloBI の相互作用エッジで表現する。

### いまのグラフを拡張するだけ(engine.build_network)
現状は `plant → insect → bird` の有向グラフ(`kind` ノード + fit 重み)で、
鳥の到来確率 = 流入エッジ(食べられる資源)の重み和。ここに **ノードの種類を足す**だけ:

- `feature`(庭に置く装置): 餌台・水場・巣箱。植物と同じく「庭の状態」。
- `mammal`(リス等) / `raptor`(Hawk等): GloBI の相互作用で庭に現れる動物。

### エッジは全部 GloBI の相互作用タイプから引く(恣意ルールを足さない)
| つながり | GloBI の型 | 例(シャーロット) |
|---|---|---|
| 餌台(種) → 食べる者 | `eats` | 種食の鳥 と **リス**(Sciurus carolinensis)が同じ種を食べる |
| リス → それを捕食する者 | `eatenBy` / `preyedUponBy` | リスの捕食者 = **Hawk**(Cooper's/Red-tailed) |
| Hawk → 獲物 | `preysOn` | 小鳥(警戒心の高い種)を襲う |
| 植物 → 昆虫 → 鳥 | `eats` 他 | 既存のまま |

→ 「餌台を開放で置く → リスが来る(eats 共有)→ リスの捕食者 Hawk が来る(eatenBy)
   → Hawk は小鳥を襲う(preysOn)→ 警戒心の高い小鳥の到来が下がる」までが、
   **すべて GloBI のエッジ**でつながる。人が書くのは「エッジの向きに従って確率が動く」枠だけ。

### 唯一の一般則(これも生態的に正統)
- **捕食の圧 = 到来確率を下げる**: Hawk(raptor)が庭にいる間、その `preysOn` 対象の鳥は
  到来確率が下がる。下げ幅は既存の **`wariness`(警戒心)特性**でスケール(臆病な種ほど強く避ける)。
  = "恐怖の景観(landscape of fear)"。恣意的な"呼ぶ/呼ばない"表ではなく、GloBI の捕食エッジ + 既存特性。
- 餌台の型(開放/かご)は「リスや大型が eats エッジを使えるか」を切り替えるだけ(物理の再現)。

### 遊びとしての帰結(餌箱ゲームの核が生態で立ち上がる)
- 開放餌台 = 早く多くの鳥が来るが、リス→Hawk 連鎖で警戒心の強い小鳥が減る。
- かご餌台/在来の実 = 来るのは遅い/地味だが、Hawk 圧が下がり多様な小鳥が居着く。
- プレイヤーは **GloBI の食物網を読んで庭を設計**する = 生態知識がそのまま攻略になる。

### 実装の順序(小さく、データが揃ってから)
1. **候補種と相互作用を GloBI から取得**(下記ヘルパー)。シャーロット優先、次点 京都。
2. 取れた `eats`/`eatenBy`/`preysOn` を data に反映(feature/mammal/raptor ノード + エッジ)。
3. engine.build_network に `feature/mammal/raptor` を足し、raptor の `preysOn` 圧を wariness でスケール。
4. まず **リス1体 + 餌台(開放/かご)** の最小連鎖だけ動かして体験を確認 → 水場・巣箱へ拡張。

> 注: GloBI はこの開発サンドボックスからは組織ポリシーで 403(遮断)。
> 実データ取得は **GloBI が通るデプロイ環境(Streamlit Cloud)** で行う。
> そのための取得ツールを開発パネルに用意した(§12)。

## 12. シャーロット拡張の材料集め(GloBI プレビュー)

`species_expand.py` + 開発パネルの「🐿️ GloBI 相互作用プレビュー」で、候補分類群の
