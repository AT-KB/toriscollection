## 1. プロダクトの全体像

### 1-1. 一行サマリ

**毎日2分、朝と夜の歯磨きの隣で、自分の庭にいる鳥たちの声に耳を澄ます儀式アプリ**。植えた植物が食物網を変え、来る鳥が変わる。Merlinが「何の鳥か」を答えるのに対し、Torisは「なぜここに来るか」を答える。

### 1-2. ターゲット顧客

- 鳥に少し興味はあるが、野鳥観察に踏み出していない
- 登山やキャンプは負担が大きい都市生活者
- 子供の頃に自然に触れた記憶を持ち、星野道夫的世界観に郷愁を覚える層
- 「鳥好き」と自認していない受動的な層が真のターゲット

### 1-3. 3つの軸

- **癒し**: 鳥の声、植物のSVG、控えめなインタラクション
- **学び**: 食物網に基づく「なぜ」の可視化、生態学リテラシー
- **集める**: 図鑑、観察記録、落とし物の3カテゴリ(小枝・羽根・羽冠)

これらは「植生で呼ぶ」というコアメカニズムの周りに配置されている。

### 1-4. 最大リスクと対応

最大リスクは「**毎日2分の継続性**」。クローズドテストでの成功定義は:
- 離脱: 最終アクセスから72時間以上経過
- 成功: 登録から7日後の時点で離脱していない(=直近3日以内にアクセスあり)

この継続性問題への解として、距離メカニクスの儀式を導入する(本書§3以降)。

- `t_fit`: 気温適合度 0〜1
- `biome_bonus`: バイオーム適合 1.0 or 0.15

## 3. これからやること - 距離メカニクスの儀式UI(JS実装)

### 3-1. 何を作るか

ホームタブの最上部に「儀式エリア」を追加する。これは**JavaScriptで動く独立したコンポーネント**で、距離メカニクスの体験を提供する。

### 3-2. 体験の核

毎日の単一行動は「**耳を澄ます**」。

1. 朝、歯磨きの隣でアプリを開く
2. 「♪ 耳を澄ます」ボタンを1回押す(ブラウザの自動再生制限の回避)
3. 3〜5羽の鳥の鳴き声が遠くから(エコーがかった音)流れ始める
4. 何もしないでいると、5秒ごとに各鳥が確率的に近づく
5. 近づくと、音量が上がり、エコーが減り、クリアになる
6. たまに鳥が逃げる(声がふっと遠ざかって消える)
7. タップなどの操作をすると、鳥は警戒して逃げやすくなる
8. 近距離まで来た鳥は「観察された」として記録される
9. アプリを閉じる、または他タブに移ると儀式終了
10. 後で図鑑を開くと、今朝の観察記録が静かに残っている

### 3-3. 5つの設計原則(交渉不能)

第一原則: **罰しない**。「逃げた」は罰ではなく自然現象
第二原則: **時間圧をかけない**。「急いで」は使わない
第三原則: **操作は最小**。耳を澄ますだけで進む
第四原則: **情報を秘める**。距離レベル数値、警戒度メーター、進捗バーは禁止
第五原則: **儚さを保つ**。「絶対に逃げない安全モード」は作らない

### 3-4. 視覚と聴覚の役割分担

- **聴覚(主)**: 距離・接近・逃走のすべてを音量・エコー・空間定位で表現。画面を見なくても体験は成立する
- **視覚(副)**: SVGの大きさ・透明度・位置で距離を示す確認役

実装の優先順位: 音優先。音が崩れたら体験は崩れる、視覚が崩れても体験は成立する。

### 3-5. 距離の4状態

| 状態 | 音声(主) | SVG表現(副) | 観察記録 |
|---|---|---|---|
| 不在 | 無音 | 描画なし | 対象外 |
| 遠距離 | 小音量、エコー強、低域強調 | 小サイズ、半透明、画面奥 | 対象外 |
| 中距離 | 中音量、エコー軽 | 中サイズ、色が見える | 対象外 |
| 近距離 | 通常音量、クリア | 大サイズ、画面手前 | **記録される** |

### 3-6. 距離遷移の確率

5秒ごとにチェック:

**操作なし(耳を澄ましている)**:
- 遠 → 中: 20%
- 中 → 近: 15% × (1 - wariness × 0.8)
- 近 → 中: 10%
- 任意 → 不在: 2〜10%

**操作あり(警戒モード)**:
- 遠 → 不在: 30%
- 中 → 不在: 40%
- 近 → 不在: 50%

### 3-7. 警戒度(wariness)の例

- スズメ 0.2 = 街中で身近、近づきやすい
- メジロ 0.4 = 庭木に来るが少し慎重
- カワセミ 0.7 = 水辺で警戒心強い
- エボシクマゲラ 0.8 = 森の奥、稀

35鳥すべての値は `STEP1_WARINESS.md` を参照。

### 3-8. 観察記録

近距離に到達した瞬間に観察記録が生成される。Finch型の罰なし蓄積。

```python
observation = {
    "bird_id": "shijukara",
    "timestamp": "2026-05-20T07:23:00",
    "duration_at_close": 12,
}
```

各鳥について累計観察回数・累計近距離滞在時間・初回観察日・直近観察日を保持。図鑑タブで「発見済 ✓ / 近距離観察 5回」のように表示。

## 4. JS実装の技術設計

### 4-1. 実装方針

**Streamlitに同居方式**(別アプリ化しない)。理由:

- 既存資産(35鳥データ、警戒度、xeno-canto、Sheets、テスター認証、図鑑、スプライト)をすべて活かせる
- 儀式以外のタブ(植える、図鑑、シミュ等)も体験の土台として必要
- データ共有が容易(別アプリだとSheetsアクセスのバックエンドが別途必要)
- JS学習の観点でも、まず1コンポーネントから始める方が安全

儀式UIは `streamlit.components.v1.html()` で埋め込む。中身はHTML/JS/CSSの大きな塊。

### 4-2. データの流れ

```
Python側 (app.py)
   ↓ render_ritual(resident_ids, biome_id, BIRDS) を呼ぶ
ritual.py
   ↓ HTMLとJSを組み立てる
   ↓ 鳥データを json.dumps() でJSONに変換
   ↓ 音源は base64 でHTMLに埋め込む
   ↓ components.html() で iframe としてブラウザに送る
ブラウザ (iframe内のJS)
   ↓ AudioContext を作成、音源を再生開始
   ↓ setInterval で5秒ごとに距離遷移
   ↓ ユーザー操作を検知して警戒モード
   ↓ SVGの大きさ・透明度を距離に応じて変える
   ↓ 観察記録をJS内部に蓄積
儀式終了時 (タブ切替・ブラウザ閉じる・30分経過)
   ↓ postMessage または localStorage で Python に送信
sheets_client.py
   ↓ observations シートに保存
```

### 4-3. Web Audio API の構成

各鳥について以下のノードを構築:

```
音源 (Audio要素) → MediaElementSource
   ↓
GainNode (音量制御)
   ↓
BiquadFilterNode (lowpass, 遠距離ほど高音域を弱める)
   ↓
DelayNode + フィードバック (エコー)
   ↓
AudioContext.destination (スピーカー)
```

距離状態が変わった時、GainNode と BiquadFilter のパラメータを `linearRampToValueAtTime` で滑らかに変化させる。

### 4-4. 自動再生制限への対処

現代のブラウザはユーザーアクション無しに `audio.play()` を許可しない。

対処: 「♪ 耳を澄ます」ボタンを最初に表示し、1回クリックされた瞬間に AudioContext を作成・音を開始する。このボタンを「儀式の開始所作」として位置づける(歯磨き前にコップを取るような自然な所作)。

### 4-5. 儀式終了の検知

以下のいずれかで儀式終了:

1. ユーザーが他のStreamlitタブをクリック(visibilitychange イベント)
2. ブラウザ/タブを閉じる(beforeunload イベント)
3. 30分経過(セーフティタイマー)

終了時、`setInterval` を停止し、観察記録を localStorage に保存。次回ホームを開いた時、JSが localStorage を読んでサーバに送信する。

### 4-6. 影響範囲

**新規ファイル**:
- `ritual.py`: 儀式UIのHTML/JS/CSS組み立て
- `observation_log.py`: 観察記録の保存メソッド

**修正ファイル**:
- `app.py`: ホームタブの最上部に儀式UIを呼び出す
- `data.py`: 各鳥に `wariness` フィールドを追加(35羽)
- `sheets_client.py`: `observations` シートへの保存関数を追加

**触らない**:
- `engine.py`, `mementos.py`, `absence_loop.py`, `xc_client.py`, `centrality.py`, `i18n.py`
- 図鑑、シミュ、落とし物、ネットワーク、使い方の各タブ
- `designbird/` の各PNG

### 4-7. 既存ホームタブとの関係

ステップ2の段階では、**ホームタブの最上部に儀式UIを追加**し、既存の中身(不在中バナー、フィールドSVG、土地選択、滞在鳥リスト、ハーモニーボタン)は**そのまま下に残す**。

ステップ5まで完成して儀式UIが十分機能した後、既存のホームタブ要素(特にハーモニーボタンや一部のフィールドSVG)を整理・削除するか、別の場所(サイドバー等)に移動するかを判断する。

2. 朝の儀式が30〜90秒に収まるか
3. 「何をすればいいか分からない」と感じる瞬間があるか
4. 鳥が近距離まで来た時、**音で気づいたか**(画面を見ずに気づけるかが重要・音中心設計の検証)
5. 2日目、3日目に自発的に開いたか

### 6-3. リクルートの原則

「鳥アプリのテスター」と呼ばない。「自然系のアプリ」「朝のリラックス用アプリ」と表現する。本当のターゲットは自分を「鳥好き」と認識していない層。

## 7. やってはいけないこと(警告リスト)

実装で絶対に取り入れない:

- ランキング、リーダーボード、フレンド機能、対戦
- スタミナ、ハート、行動回数制限
- ガチャ、ルートボックス
- 「ストリークが切れます」「逃げました」などの罪悪感ベース通知
- 進捗バー、距離レベル数値、警戒度数値の表示
- 時間制限イベント
- 安全モード(逃げない設定)
- 接近を促す通知(「急いで!」)
- 距離レベルアップの演出

これらを入れるとTorisは別のアプリになる。

## 8. 想定される技術的な壁

### 8-1. 自動再生がどうしても動かない

対処: ユーザーアクション(ボタンクリック)直後のplay()で動くか確認。ブラウザのバージョン・OSをチェック。最悪、シンプルな確認だけ通してステップ4で本格対処。

### 8-2. components.htmlがStreamlitの再実行で消える

Streamlitはユーザーの操作で再実行されるが、components.html内のiframeはStreamlit側の操作には影響されない(条件付き)。
対処: 儀式中はサイドバー操作を控えてもらう設計。注意書きは出さない(体験を壊す)。

### 8-3. スマホでの動作が不安定

iOS Safariは特にWeb Audioの挙動に癖がある(サイレントモード時、画面ロック時)。
対処: PC Chromeで動作確認後、スマホでチェック。スマホで動かない箇所は個別対処。

### 8-4. 観察記録の保存

postMessageからPython→Sheetsへの経路はStreamlitで少し癖がある。
対処: まずlocalStorageに保存し、次回ログイン時にJSが読み出してPython側にユーザーアクション経由で送る。リアルタイム送信である必要はない。

## 9. 進め方の規律(将来のAI/開発者へ)

1. **ユーザーはJS初心者**。コードだけでなく、何が起きているかの説明を都度添える
2. **ステップは1つずつ進める**。複数ステップを同時に実装して大きな差分を出さない
3. **動作確認を必ず挟む**。コードを書いたらまずデプロイしてもらい、確認結果を聞いてから次に進む
4. **判断事項は `RITUAL_IMPLEMENTATION_PLAN.md` §3 を参照**。すでに確定済みのものを蒸し返さない
5. **仕様書 `DISTANCE_MECHANIC_SPEC.md` の音中心原則を絶対守る**。視覚優先に倒さない
