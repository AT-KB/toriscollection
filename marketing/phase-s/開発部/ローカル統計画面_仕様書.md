# アプリ内ローカル統計画面 — 仕様書(Phase S / 開発部)
**根拠**: `toris_ceo_kettei_memo_phase_s.md` 決定2c・Phase S完了定義#5。
**位置づけ**: 分析のサーバー基盤を作らず(決定2)、**端末内にのみ記録し、ユーザー操作でのみ書き出せる**統計画面。原則1(受動性・ローカル保存の思想・ダークパターン禁止)と完全両立させ、同時に**スクショ共有の自然な種**にする。
**本書のDoD**: この仕様書をCEOが承認(実装完了はPhase 1条件で可)。

---

## 1. 設計原則(交渉不能原則との整合)
- **端末内のみ**。サーバー送信ゼロ。既存のローカル保存(セーブコード/`st.session_state`/localStorage)の枠内に収める。
- **書き出しはユーザーの明示操作のみ**(自動送信・裏送信をしない)。
- **数字で急かさない**。連続日数などは「途切れても罰しない」表現(原則2)。ストリークを煽ってデイリーを強制しない。
- **かわいく・静かに**。図鑑や庭と同じトーン。

## 2. 表示する指標(すべて既存のローカルデータから導出)
実装時に現行コードのフィールド名を要確認(下記は本セッションで確認した想定):
| 指標 | 由来(既存データ) | 備考 |
|---|---|---|
| 出会った鳥の数 / 図鑑進捗 | `discovered` 集合・`observed`(bird_id→count/first/last) | 「◯ / 37 種」 |
| 近くで観察した回数 | `observed[bird]["count"]` | 鳥ごと/合計 |
| 会った日数(のべ) | `_mark_met_today` 由来の日次カウンタ | 原則2に沿い「減らない」旨を添える |
| 連続日数(任意・控えめ) | 上記日次記録から算出 | **途切れても咎めない**文言。強制しない |
| ラジオ再生秒数(**新規計測**) | ラジオ iframe の再生時間を積算しローカル保存 | 活性化判定に使用 |
| 集めた落とし物 | `mementos` | 「羽根/小枝/羽冠」 |
| 今の庭の豊かさ | 滞在中の鳥・植えた植物・ネットワークのリンク数 | 既存 `residents`/`planted`/network stats |

## 3. 活性化(activation)の定義と記録
決定2の定義をローカルカウンタで実装:
> **初回セッション内で `new_bird_first_view`(新しい鳥の初見)かつ `birdsong_play_seconds ≥ 5`(鳴き声を5秒以上再生)に到達 = 活性化。**
- ローカルにフラグ `activated: bool` と到達日時を1回だけ記録。サーバー送信しない。
- `birdsong_play_seconds` を新規に積算(ラジオ/個別再生の実再生秒。**新規実装が必要な唯一の計測**)。
- この値は週次マイクロサーベイ(決定2b)や、ユーザーが「統計を共有」した時のスクショにのみ現れる。

## 4. 「統計を共有する」ボタン(ユーザー操作のみ)
- 統計画面に**共有用の1枚(かわいい要約カード)**を生成し、**ユーザーが押した時だけ**書き出す。
- 手段: (a) 画面のスクショを促す静止カード表示 /(b) 可能なら画像書き出し。**サーバーへは一切送らない**。
- カードに載せる例: 「◯ birds met · ◯ days in the garden · your radio has ◯ voices」+ ドット絵の鳥数羽。**出典・数字は誠実に**、煽らない。
- ねこあつめ型の自然発生的スクショ共有を狙う(共有カード"機能"の作り込みは最小限。まずはテスターが自然に撮るかを観察=決定メモ§2-4開発成果物Bと整合)。

## 5. 実装メモ(開発部向け・Phase 1で着手)
- 既存の図鑑/落とし物/会った日数のロジックを**再利用**し、新規は `birdsong_play_seconds` の積算と `activated` フラグのみ。
- 保存はセーブコード/ローカルに収める(既存のシリアライズに1〜2キー追加)。**「壊さない検証」**(構文→AppTest起動時例外0→tests全pass→ラジオJSは Node --check + Playwright)を必ず通す。
- **Renderは Python 3.11**。f-string 式内バックスラッシュ禁止(既知インシデント。push前にAST検査)。
- UIは新タブ追加ではなく、既存の該当箇所(図鑑/サイドバー)への小さな追加でも可。過剰な新規UIは避ける。

## 6. 非対象(Phase Sではやらない)
- サーバー送信イベント(`app_open_after_gap` 等)は**凍結**(決定2)。
- 外部分析基盤(TelemetryDeck等)は**Phase 1ゲートで判断**。Phase S中は調査チケットのみ。
- 共有カードの作り込み・共有導線の新規開発は最小限(自然発生の観察が先)。
