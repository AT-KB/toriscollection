## 交渉不能の原則(5つ、`00_共通サマリ.md` 準拠)

すべての部署・すべての提案は、以下5つに違反してはならない。違反する提案は、
どの部署が発案したものであっても差し止め、CEOにエスカレーションする。

1. **受動的である** — 急かす設計(スタミナ・時短課金・デイリー強制)は入れない。
2. **罰しない** — 撹乱(嵐・伐採等)で庭が痩せても、コレクション(図鑑・会った日数)は減らない。
3. **鳥の声と癒しは常に無料** — 「課金しないと聴けない」は絶対にやらない。
4. **生態に誠実** — 関係(食物網)は GloBI ベース。恣意的な"なぜ来たか"を足さない。
5. **かわいさ最優先** — 来る瞬間・鳴き声・ドット絵の"かわいさ"を最優先に守る。

## 部署とサブエージェント

- **企画部**(`kikaku`): 体験設計・機能仕様・数値バランス・ロードマップ。コードは編集しない。
- **SNS部**(`sns`): 発信・ASO・コラボ文面。コードには一切触れない。
- **開発部**(`dev`): 実装・データ・テスト・デプロイ。変更ごとに「壊さない検証」
  (構文チェック→AppTestで起動時例外0→tests全pass→ラジオJSはNode `--check`+Playwrightで実挙動確認)
  を必ず実行する。詳細は `toris_collection/docs/team/03_開発部.md` を参照。

定義は `.claude/agents/kikaku.md` / `.claude/agents/sns.md` / `.claude/agents/dev.md`。

## 承認が必要な変更

コード・データ構造・仕様を大きく変える作業は、いきなり実行せず、まず計画を提示して
CEO(ユーザー)の承認を待つこと。何が「大きく変える」に該当するかの具体的な基準は
`toris_collection/docs/team/05_運用ルール.md` §1 を参照。交渉不能の原則(上記5つ)に触れる提案は、
規模の大小に関わらず必ずCEO確認を要する。

## 作業開始前の運用ルール(リモート確認)

**どんな作業を始める前にも、必ず最初に `git fetch` でリモート(GitHub)の最新状態を確認する。**
`git status` や `git log --oneline -5` だけではローカルの `main` が `origin/main` より
遅れていても気づけない。フェッチ後、ローカルが遅れていることが分かった場合は、
作業内容を検討・実行する前に **先に `git pull`(fast-forward)でローカルを最新化**してから
着手する。ローカル側に未コミットの変更がある場合は、`git status` で確認し、
必要なら `git stash -u` で退避してから pull し、作業後に戻す。

この手順を怠ると、リモートに既にある正式資料(例: `toris_collection/docs/team/`)に気づかず、
別の場所に重複・矛盾する成果物を作ってしまう恐れがある(実際に一度発生した)。
迷ったら「まず fetch、遅れていたら pull」を徹底する。
